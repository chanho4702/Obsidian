# 🎯 내 목표 — MSA Starter Template

> [!abstract] 한 문장
> **새 서비스 하나를 추가하면 인증·게이트웨이·관측·배포가 자동으로 붙는, 확장 가능한 MSA 스타터 템플릿**을 직접 만들고 그 위에 실제 서비스(PMS)를 올린다.
> 성장 경로: **Docker Compose → Kubernetes + ArgoCD → Spring AI**

표준화된 환경에서 빠르게 시작하고, 공통 모듈을 재사용하며, 인프라를 자유롭게 교체·확장할 수 있는 MSA 스타터 템플릿. 로컬 개발(Compose)부터 운영 배포(K8s), AI 확장까지 단계적으로 성장하도록 설계한다.

## 🏗️ 아키텍처 개요

요청 흐름:

```
Client → Nginx → API Gateway(Spring Cloud Gateway) → MSA Services
                       ↑ JWT 검증
                   Keycloak (OAuth2 / OIDC / SSO)
```

인증은 **Keycloak**이 담당하고, **Gateway 단에서 JWT를 검증**한다. (검증 위치 결정 → [[phase0-week-breakdown|Phase 0 W3]])

## 🧩 핵심 구성 요소

### Gateway & 인증
| 구성 | 역할 |
|---|---|
| **Nginx** | Reverse Proxy, Static Files, SSL Termination |
| **API Gateway** (Spring Cloud Gateway) | Routing, Filter, Rate Limiting, Circuit Breaker, Logging |
| **Keycloak** | OAuth2 / OIDC, SSO, JWT, RBAC |

### 공통 모듈 (Common Modules)
| 모듈 | 역할 |
|---|---|
| `starter-core` | 공통 응답(ApiResponse)·ErrorCode·GlobalExceptionHandler |
| `starter-security` | 보안 / 인증 / 인가 |
| `starter-logging` | 로그 / MDC / traceId 추적 |
| `starter-domain` | 공통 도메인 / 유틸 |

> 공통 응답·예외 표준화의 실제 구현 레퍼런스 → [[arms-engine-재사용패턴]]

### 인프라 (Infrastructure)
- **Data** — MySQL(관계형) · Redis(Cache/Session) · Kafka *(이벤트, Phase 4)* · OpenSearch *(검색/분석, Phase 4)*
- **Secret** — HashiCorp Vault (DB 비번·API Key·JWT Secret) *(Phase 3에서 `.env` → Vault 이관)*
- **Cross-Cutting** *(후속)* — Config Server · OpenTelemetry · Loki · Istio *(선택)*
  - ※ Service Discovery(Eureka)는 **도입하지 않음** — K8s가 제공하므로 중복 회피 ([[phase5-week-breakdown|Phase 5]])

### 관측 & 모니터링 (Observability)
- **Spring Actuator** → **Prometheus**(metrics) → **Grafana**(대시보드) → **Alertmanager**(알림)
- 분산 추적: **OpenTelemetry + Loki** ([[phase3-week-breakdown|Phase 3]])

### DevOps & CI/CD
```
GitHub Actions → 빌드/테스트 → SonarCloud(품질 게이트) → 멀티스테이지 빌드 → GHCR
```
- Compose 단계 배포: `docker compose pull`로 GHCR 이미지 사용 (별도 CD 불필요)
- 운영 배포(CD): **ArgoCD GitOps** ([[phase5-week-breakdown|Phase 5]])
- 상세 → [[phase1.5-cicd]]

## 🚀 단계별 로드맵 (요약)

> 전제: 직장 병행 **주 10~15시간**, 전체 약 6~7개월 (2026-06 중순 ~ 2027-01 초). 상세·주차별 DoD는 [[msa-roadmap]].

| Phase | 기간 | 목표 | 산출물 체크포인트 |
|---|---|---|---|
| **0** 인증 마무리 | W1–3 (6/15~7/5) | OAuth2/OIDC → Keycloak → Gateway 검증 | Gateway에서 Keycloak 토큰 검증 통과/실패 동작 |
| **1** 템플릿 골격 ⭐ | W4–7 (7/5~8/2) | 새 서비스가 바로 붙는 구조 | `docker compose up` 한 방으로 전체 스택 기동 **(1차 목표)** |
| **1.5** CI | +3일 | 자동 빌드/이미지 푸시 | main 머지 시 GHCR에 이미지 |
| **2** DB 실전 | W8–12 (8/2~9/6) | 인덱스·트랜잭션·N+1 | 느린 쿼리 인덱스 개선 + N+1 제거 측정치 |
| **3** 운영 인프라 | W13–16 (9/6~10/4) | 관측 가능성 확보 | traceId로 전 서비스 분산 추적 |
| **4** 이벤트 확장 | W17–20 (10/4~11/1) | 서비스 간 직접 호출 줄이기 | Outbox + DLQ로 신뢰성 있는 이벤트 처리 |
| **5** K8s + ArgoCD | W21–26 (11/1~12/13) | 오케스트레이션 + GitOps | Git 푸시로 무중단 배포 |
| **6** Spring AI | W27–28 (12/13~1/3) | LLM·RAG 연동 | RAG 기반 문서 Q&A |

**진행 순서 한 줄:** 인증 마무리 → MSA 템플릿 골격 → DB 실전 → 운영 인프라 → 이벤트(Kafka) → K8s/ArgoCD → Spring AI

## 🪜 두 가지 원칙

1. **이론은 just-in-time.** 책을 다 떼고 시작하지 말고, 템플릿에 실제 도메인을 붙여 문제를 만났을 때 그 주제를 판다. (느린 쿼리 → 그때 인덱스/실행계획)
2. **각 주는 DoD(완료 기준)로 끝낸다.** "읽었다"가 아니라 "돌아간다"로 판정. DoD 미통과 시 다음 주로 넘어가지 않는다.

---

## 🧱 템플릿 위에 올릴 것 — PMS

이 템플릿은 그 자체가 목적이 아니라 **플랫폼**이다. 완성 후 첫 번째 실제 서비스로 **PMS(프로젝트 관리 시스템)**를 올린다.
- 순서: **템플릿(플랫폼) 먼저 → PMS 나중**
- 템플릿엔 인증 뼈대(Keycloak Realm/Client/Role + Gateway JWT 검증)와 RBAC "꽂히는 자리"만 만들고, 실제 권한 모델(PROJECT_ADMIN/PM/LEADER/DEVELOPER/VIEWER)은 PMS 단계에서 → [[pms-backlog]]
