## 0. 먼저 개념을 내 것으로 — 비유 5개

어려운 게 당연하다. 용어가 5개 겹쳐 있어서 그렇다. 비유로 한 번에 정리:

| 용어 | 비유 | 한 줄 정의 |
|---|---|---|
| **RT 회전(rotation)** | 쓸 때마다 새 표로 바꿔주는 영화표 | refresh token은 일회용 — 쓰면 새 걸로 교체되고 옛 건 폐기 표시 |
| **재사용 탐지** | 폐기된 옛 표가 또 나타남 = 누가 복사했다는 물증 | 정상 세계에선 옛 토큰이 다시 올 수 없다. 왔다면 복사본(도난)이 존재한다 |
| **가족(family) 폐기** | 그 복사표에서 파생된 표 전부 무효화 | 도둑과 진짜 사용자를 구별할 수 없으니 그 로그인에서 나온 토큰을 전부 죽인다. 진짜 사용자는 재로그인하면 됨 |
| **조건부 UPDATE 원자 선점** | 선착순 1명 도장 — 창구(DB)가 딱 한 명에게만 찍어줌 | `UPDATE ... WHERE revoked=false` — 동시에 두 요청이 와도 DB가 정확히 하나만 성공시킨다 |
| **grace 30초** | "방금 낸 표가 또 왔네? 두 탭이 동시에 눌렀구나" 하고 봐주기 | 교체 후 30초 이내 재사용은 도난이 아니라 멀티탭 경쟁으로 판정 — 오탐(정상 사용자 강제 로그아웃) 방지 |

**한 문장으로 전체 요약** (이거 하나만 외워도 된다):
> "refresh token을 일회용으로 만들면 도난을 탐지할 수 있는데, 그 탐지 로직이 **동시 요청**과 **트랜잭션 롤백**이라는 두 지점에서 무너질 수 있다는 걸 발견했고, 원자 선점 + grace + noRollbackFor로 고친 뒤 실제 Postgres에서 멀티스레드 테스트로 증명했습니다."

## 1. 미끼 ① — "동시성에서도 성립하는 재사용 탐지"

### 1분 답변 스크립트

> "refresh token 회전은 옛 토큰 재사용을 도난 신호로 쓰는 표준 패턴인데, 구현을 뜯어보니 '폐기됐는지 확인'과 '폐기 표시' 사이가 원자적이지 않았습니다. 같은 토큰으로 동시에 요청 2개가 오면 둘 다 통과해서 살아있는 토큰이 2개가 되고 — 공격자가 진짜 사용자와 같은 순간에 refresh하면 탐지가 조용히 우회됩니다. 반대로 브라우저 멀티탭에서는 늦은 탭이 도난으로 오판돼 정상 사용자가 강제 로그아웃당하고요.
> 그래서 폐기 표시를 `UPDATE ... WHERE revoked=false` 조건부 UPDATE 한 방으로 바꿔 DB가 딱 한 요청만 성공시키게 했고, 진 쪽은 '교체된 지 30초 이내면 멀티탭 경쟁으로 관용, 지났으면 도난'으로 분기했습니다. Auth0 같은 상용 IdP도 같은 leeway 방식을 씁니다.
> 검증은 Testcontainers로 실제 Postgres를 띄워서, 같은 토큰으로 2스레드 동시 refresh를 걸면 정확히 1승 1패가 나오는지 테스트로 확인했습니다."

### 꼬리질문 방어 (5세트)

1. **어떤 문제였나?** — check-and-set 비원자성. READ_COMMITTED에서 두 트랜잭션이 둘 다 `revoked=false`를 읽고 통과 → 탐지 우회(공격) + 멀티탭 오탐(UX 사고) 두 얼굴.
2. **왜 조건부 UPDATE였나? 락은?** — 비관적 락(`PESSIMISTIC_WRITE`)도 검토. 락은 유저 조회~INSERT까지 행 락을 쥐고 있어 경합 시 대기가 생기고, 락만으로는 "진 쪽을 어떻게 처리할 것인가"(오탐 문제)가 그대로 남아 grace 분기가 어차피 필요. 조건부 UPDATE는 락 없이 영향 행 수(1 or 0)로 승패가 즉시 판명.
3. **역할 범위는?** — 1인 프로젝트. 분석 문서 작성 → 설계 → 구현 → 테스트까지 전부. (AI 도구를 활용했지만 설계 결정·검증 기준은 직접 정의 — 물어보면 솔직하게: "AI 페어로 구현 속도를 냈고, 저는 설계 결정과 검증에 집중했습니다")
4. **결과를 어떻게 확인했나?** — 실 PG 2스레드 테스트: `CyclicBarrier`로 동시 출발 → Rotated 1개 + ConcurrentRotationException 1개 + DB에 미폐기 토큰 정확히 1개 단언. H2로는 락 동작이 달라 의미가 없어서 Testcontainers를 도입했다.
5. **다시 한다면?** — 처음부터 "보안 기능은 경쟁 조건까지가 스펙"이라는 관점으로 설계했을 것. 그리고 메트릭(reuse_detected 카운터)을 같이 넣어 운영 알림까지 연결했을 것(현재는 WARN 로그까지).

### 함정 질문 대비

- **"grace 30초면 공격자도 30초 안에 쓰면 통과잖아요?"** → 맞다, 의도된 트레이드오프. grace 내 공격자는 토큰 1회 연장을 얻지만 가족 폐기를 피할 뿐 탐지 체계 자체는 유지되고, 오탐(정상 사용자 로그아웃)의 확실한 피해와 교환한 것. Auth0 등도 같은 선택.
- **"쿠키는 왜 안 지우나요?"** → 경쟁 패배 응답에서 쿠키를 지우면 승자 탭이 방금 심은 **새 쿠키**를 지워버린다. 쿠키는 브라우저 공유라 패자 탭은 새 쿠키로 재시도하면 성공.

## 2. 미끼 ② — "예외 롤백이 보안 로직을 무력화하던 버그"

### 1분 답변 스크립트

> "도난을 탐지하면 가족 전체를 폐기하고 예외를 던지는 구조였는데, 그 예외가 RuntimeException이라 스프링 `@Transactional`이 **방금 한 가족 폐기까지 롤백**시키고 있었습니다. 즉 운영에서는 도둑을 잡아도 401만 주고 실제 폐기는 DB에 남지 않아서, 도둑이 가진 살아있는 토큰이 계속 유효했던 거죠.
> 더 흥미로운 건 기존 테스트가 통과하고 있었다는 점입니다. `@DataJpaTest`는 테스트 전체를 한 트랜잭션으로 묶어서 롤백 전까지는 변경이 보이거든요. 슬라이스 테스트로는 **원리적으로 잡을 수 없는** 버그였습니다.
> `noRollbackFor`로 고치고, 실제 Postgres에서 예외를 받은 뒤 **별도 트랜잭션으로** 다시 조회해 폐기가 커밋됐는지 확인하는 테스트를 추가했습니다. '보안 기능이 있다'와 '트랜잭션 경계에서도 성립한다'는 다른 문제라는 게 교훈입니다."

### 꼬리질문 방어 (5세트)

1. **어떤 문제였나?** — `revokeFamily()` 후 `ReuseDetectedException` throw → 언체크 예외는 기본 롤백 → 폐기가 취소. 탐지의 "잡았다" 기록이 증발.
2. **왜 noRollbackFor였나?** — 대안은 REQUIRES_NEW로 폐기를 별도 트랜잭션 분리. noRollbackFor가 더 단순하고, 이 트랜잭션에서 폐기 외에 보존할 다른 쓰기가 없어 부작용 없음.
3. **어떻게 발견했나?** — 코드 리뷰 중 "이 예외가 던져지면 직전 UPDATE는 어떻게 되지?"를 따라가다 발견. 테스트가 green인데도 운영 동작이 다를 수 있다는 의심이 출발점.
4. **결과 확인은?** — 실 PG 테스트: 도난 시나리오 실행 → 예외 확인 → **새 트랜잭션**에서 `findAll()` → 전 토큰 revoked 단언. `@SpringBootTest`(비트랜잭션)라 조회가 진짜 별도 트랜잭션인 게 포인트.
5. **다시 한다면?** — 보안 이벤트 기록(폐기·감사 로그)은 처음부터 "예외 경로에서도 커밋되는가"를 테스트 항목에 넣는다. 슬라이스 테스트의 트랜잭션 래핑을 맹신하지 않는다.

## 3. 미끼 ③ — "docker split-horizon OIDC"

### 1분 답변 스크립트

> "컨테이너로 올리니 로그인이 깨졌는데, 원인은 브라우저는 `localhost:8080`으로, 컨테이너는 내부 DNS `keycloak:8080`으로 **같은 Keycloak에 서로 다른 주소로 접근**한다는 것이었습니다. Keycloak은 토큰의 발급자(iss)를 브라우저가 접근한 호스트 기준으로 찍는다는 걸 E2E로 확인했고, 그래서 스프링의 자동 디스커버리가 어느 쪽 URL로도 성립할 수 없었습니다.
> 해법은 ClientRegistration을 수동 구성해서 '브라우저가 가는 URL'과 '서버끼리 통신하는 URL'을 분리하는 것 — 인가 요청과 iss 검증은 localhost 기준, 토큰 교환과 JWKS는 내부 DNS 기준으로요. 'URL은 하나'라는 가정이 컨테이너에서 깨진다는 걸 배웠습니다."

### 꼬리질문 방어 — 짧게

1. **문제**: OIDC 디스커버리는 문서의 iss와 실제 토큰 iss가 일치해야 하는데, 이중 호스트 환경에선 불일치.
2. **왜 수동 구성**: 프로퍼티만으로는 URL을 용도별로 쪼갤 수 없음. `@Profile("docker")` 빈이 있으면 Boot 자동 디스커버리가 백오프.
3. **확인**: docker compose 전체 스택에서 로그인 E2E 통과.
4. **다시 한다면**: 처음부터 리버스 프록시로 호스트를 통일하는 안(외부에서도 같은 호스트)과 비교 검토했을 것 — 다만 로컬 개발 편의는 현재 안이 우위.
