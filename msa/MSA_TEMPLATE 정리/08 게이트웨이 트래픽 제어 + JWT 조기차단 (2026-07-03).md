## 아키텍처 결정 2가지 (중요)

**1. 이중 검증(zero-trust)** — 게이트웨이 검증은 1차 방어(쓰레기 트래픽 조기 차단)일 뿐, board-service·auth-server의 자체 JWKS 검증(최종 방어)을 대체하지 않는다. 내부망에서 게이트웨이 우회 직접 호출이 있어도 안전.
→ **동기화 규칙**: board-service에 공개 경로를 추가하면 gateway `SecurityConfig`에도 추가해야 한다(게이트웨이가 더 좁으면 먼저 401을 내버림).

**2. CORS를 Security 빈으로 이전한 이유** — Security가 내는 **401 응답에도 CORS 헤더가 붙어야** 브라우저의 fetch가 상태코드를 읽고 myFront의 401→refresh 흐름이 산다. yml globalcors는 라우팅 단계라 Security 401에는 헤더를 못 붙인다. 회귀 테스트(`unauthorizedResponseCarriesCorsHeaders`)로 고정.

## 최종 리뷰에서 잡힌 실동작 결함 (교훈)

- **C1 (Critical)**: CircuitBreaker를 달면 **Resilience4j 기본 TimeLimiter 1초**가 라우트 전체를 캡한다 — 1초 넘는 '정상' 응답까지 fallback 503으로 잘리고, 선언해둔 response-timeout 10s가 무효가 됨. 1.5초 지연 정상 스텁(JDK HttpServer) 테스트로 RED 재현 → `resilience4j.timelimiter.instances.board.timeout-duration: 11s`로 수정. **CB 도입 시 TimeLimiter 명시는 필수 체크리스트.**
- I1: Retry 기본 exceptions에 TimeoutException이 포함돼 있어 타임아웃도 재시도 → fallback 지연 3배 증폭 가능. `exceptions: java.io.IOException`(연결 실패만)으로 한정.
- I2: Redis 없이 돌리면 rate limit 경로 요청마다 `Error calling rate limiter lua` **ERROR 로그가 남는 게 정상**(fail-open) — README에 명시.
