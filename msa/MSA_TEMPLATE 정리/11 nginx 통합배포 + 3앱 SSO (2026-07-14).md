## 무엇을 만들었나

프론트가 3개가 됐다 — myFront(포털, 인증 연동 완료), wiki-front(컨플루언스 클론 MVP), alm-front(지라 클론 MVP). 이걸 **nginx 하나의 오리진(`http://localhost`)으로 통합 서빙**하고, 기존 인증 플랫폼(Keycloak + auth-server BFF)에 물려 **한 번 로그인하면 3앱 전부 인증되는 SSO**를 완성했다.

```
브라우저 ── http://localhost (nginx, docker)
              ├─ /            → myFront/dist   (정적, /srv/portal)
              ├─ /wiki/       → wiki-front/dist (/srv/apps/wiki)
              ├─ /alm/        → alm-front/dist  (/srv/apps/alm)
              └─ /api/**, /oauth2/**, /login/**, /.well-known/**
                              → host.docker.internal:8000 (gateway)
                                  ├─ auth-server (lb://, :9000)
                                  └─ board-service (lb://)
Keycloak :8080 (기존 그대로 — 브라우저가 직접 접근)
```

> [!important] SSO의 핵심은 코드가 아니라 오리진 설계
> RT 쿠키가 host-only(`localhost`) + `path=/api/auth`이므로, **오리진이 하나면 3앱이 쿠키를 자동 공유한다.** CookieFactory는 한 줄도 안 고쳤다. 서브도메인 방식이었으면 쿠키 Domain 확장 + hosts + CORS 작업이 전부 필요했다.

## returnTo 쿠키 — "원래 SSO도 이렇게 하나?"에 대한 답

문제: 로그인 여정이 **브라우저 → auth-server → Keycloak → auth-server**의 풀 리다이렉트라 SPA 메모리가 전부 날아간다. `/wiki/pages/3`을 보다가 로그인한 사람이 myFront `/app`에 떨어지면 안 된다.

패턴 자체는 업계 표준이고 이름만 다르다:

| 구현체 | 메커니즘 |
|---|---|
| 구글 | `continue=` 쿼리 파라미터 (자기 도메인으로만 제한) |
| GitHub | `return_to` 쿼리 파라미터 |
| SAML | **RelayState** — 아예 프로토콜 스펙에 있는 공식 필드 |
| Auth0 SDK | `appState.returnTo`를 `state` 파라미터에 탑재 |
| NextAuth | `callbackUrl` **쿠키** ← 우리와 동일 |
| Spring 기본 | 세션 saved request (HttpSessionRequestCache) |

우리가 쿠키를 고른 이유: SPA가 직접 `/oauth2/authorization/keycloak`으로 이동하는 구조라 Spring saved-request가 잡을 "가로챈 원래 요청"이 없고, 그 엔드포인트는 Spring 관리라 임의 쿼리 파라미터를 끼우기 애매하다.

동작:
1. 프론트가 로그인 리다이렉트 **직전**에 `post_login_redirect=/wiki/pages/3; path=/; max-age=300; SameSite=Lax` 쿠키를 심는다 (값 인코딩 안 함 — 라우트 경로에 쿠키 금지 문자 없음)
2. `LoginSuccessHandler`가 읽고 **검증**: `/`로 시작 AND `//`로 시작 금지 (오픈 리다이렉트 방어 — `//evil.com`은 브라우저가 절대 URL로 해석하는 트릭)
3. 통과하면 `frontendUrl + path`로 리다이렉트, **무효 값이어도 쿠키는 소비·삭제**(일회용), 실패/부재 시 `/app` 폴백

테스트 4개로 계약 고정: 유효 경로 왕복 / 부재 폴백 / 절대 URL 거부 / `//` 거부.

### AuthGate 설계 (wiki/alm 공통)

- 마운트 시 `tryRefresh()` → 성공: `fetchMe()` 후 children 렌더 / 실패: returnTo 쿠키 심고 `location.assign(loginUrl())`. KC 세션이 살아 있으면 이 왕복이 **무프롬프트** — 이게 SSO 체험.
- `enabled = import.meta.env.PROD` 기본 — vite dev/vitest에서는 게이트 꺼짐. 인증 검증은 nginx 프로덕션 경로로만 (dev 프록시 배선은 YAGNI로 제외).
- **myFront와 의도적으로 다른 계약**: `useAuth`가 Provider 밖에서 throw하지 않고 `{user:null, logout:noop}` 반환. 기존 테스트 138개(77+61)가 App을 게이트 없이 렌더하기 때문. myFront는 Provider 필수·throw 유지.

### nginx conf 핵심

- 정규식 location `~ ^/(api|oauth2|login|\.well-known)/`가 프록시 담당. **뒤에 슬래시를 요구**하므로 bare `/login`(myFront SPA 라우트)은 안 잡히고 `/login/oauth2/code/keycloak`(OIDC 콜백)만 프록시된다 — 한 글자 차이로 라우팅이 갈리는 지점.
- SPA 폴백은 `alias` 대신 `root` 사용: `location /wiki/ { root /srv/apps; try_files $uri /wiki/index.html; }` — alias+try_files의 고전 버그 회피.
- `= /wiki { return 301 /wiki/; }` 정규화로 슬래시 없는 진입이 포털 폴백에 삼켜지는 것 방지.
- 마운트는 중첩 바인드 금지 구조: `/srv/portal`과 `/srv/apps/{wiki,alm}` 분리 (myFront dist 안에 wiki를 마운트하면 호스트 dist에 빈 디렉토리가 생기는 부작용).

## 리뷰 루프가 잡은 실버그 3건 (교훈)

> [!warning] ① AuthGate 무한 refresh 루프 — React effect 의존성 함정
> 플랜 코드의 `redirect = (url) => window.location.assign(url)` prop **기본값은 렌더마다 새 함수**다. 이게 effect 의존성 `[enabled, client, redirect]`에 들어가면: 로그인 성공 → `setUser`(새 객체) → 재렌더 → 새 redirect 참조 → effect 재실행 → 다시 refresh/me → … **프로덕션에서만** 터지는 무한 네트워크 루프(dev는 게이트 꺼짐). 회귀 테스트로 RED 재현("expected 1 times, but got 2 times") 후 모듈 레벨 `defaultRedirect` 상수로 수정. **교훈: 함수 타입 prop에 인라인 기본값 + effect deps 조합은 금물. 기본값은 모듈 상수로.**

> [!warning] ② nginx `$server_port`는 거짓말을 한다
> `proxy_set_header X-Forwarded-Port $server_port`는 컨테이너 **내부** listen 포트(80) 고정값이다. Docker 포트 매핑은 프로세스에게 투명해서 `NGINX_PORT=8080`으로 띄워도 nginx는 80만 안다. Spring은 X-Forwarded-Port를 우선하므로 redirect_uri 포트가 틀어진다. 수정: 헤더 제거 — 포트는 `X-Forwarded-Host: $http_host`(브라우저 Host 헤더, 비표준 포트 자동 포함)로만 전달.

> [!warning] ③ `down -v`가 board-service를 벽돌로 만든다
> README가 안내하는 realm 재import(`down -v && up -d`)는 DB 볼륨을 날리는데, init 스크립트가 `authdb`만 만들고 `boarddb`는 안 만들었다(과거에 수동 `CREATE DATABASE`로 만들었던 것). 재기동 후 board-service 부팅 실패. `init-authdb.sql`에 `CREATE DATABASE boarddb;` 한 줄 추가로 해결. **교훈: "문서가 안내하는 리셋 경로"는 문서 작성 시점에 실제로 밟아봐야 한다.**

## 교차 repo 와이어 계약 (최종 리뷰에서 5종 전부 ✅)

1. **쿠키 계약**: 이름 `post_login_redirect`·`path=/`·raw 값 — 4개 repo 일치 (wiki/alm returnTo.ts는 blob 동일)
2. **리다이렉트 체인**: AuthGate가 저장하는 `pathname+search`는 basename 포함 풀 경로라 서버 검증(`/` 시작)을 항상 통과
3. **nginx ↔ gateway 라우트 패리티**: 프록시되는 모든 경로에 gateway 라우트 존재, bare `/login`은 SPA로
4. **realm redirectUris**: `http://localhost/login/oauth2/code/keycloak` 등록 — 단, **`:8000` 항목은 삭제 금지**(게이트웨이가 X-Forwarded-Port를 append하면 콜백이 :8000으로 탈 수 있는 폴백)
5. **vite base ↔ basename ↔ nginx try_files ↔ compose 마운트**: 앱별 3중 일치

## 남은 것

**수동 브라우저 E2E** (파괴적/대화형이라 자동화 제외) — [[2026-07-14 nginx 통합배포+SSO 수동 E2E 체크리스트]]:
1. 프론트 3개 빌드 → 2. `docker compose down -v && up -d` (⚠ DB 초기화) → 3. 백엔드 4개 기동, auth-server만 `FRONTEND_URL=http://localhost` → 4. `/wiki` 진입→로그인→복귀, `/alm` 무프롬프트, 로그아웃 전파

**follow-up (수용된 minor)**:
- AuthGate: `tryRefresh` 성공 후 `fetchMe` reject 시 스피너 고착 — try/catch로 로그인 리다이렉트 폴백 (wiki·alm 2파일)
- nginx: 301 쿼리스트링 보존(`$is_args$args`), proxy keepalive(http/1.1)
- `.wiki-auth-user`/`.jira-auth-user` 클래스 스타일링 (디자인시스템 터치 때)
- 게이트웨이 X-Forwarded-Port append 여부 E2E 실측

**알려진 리스크 (다음 하드닝 후보)**: 다중 탭 동시 refresh — wiki·alm 탭이 동시에 열리면 같은 RT로 동시 refresh → 뒤진 쪽이 superseded 토큰 사용 → 재사용탐지가 도난으로 오인 → 패밀리 폐기(전체 로그아웃). 정석 처방은 auth-server에 superseded 토큰 유예기간(~30초).
