## 1. 왜 이런 아키텍처인가 — MSA와 우산 repo

> [!note] 배경지식 — 마이크로서비스와 "템플릿"의 의미
> 마이크로서비스 아키텍처(MSA)는 기능 단위로 서비스를 쪼개 **독립 배포·독립 장애 격리**를 얻는 대신, 인증·라우팅·서비스 발견·관측 같은 **횡단 관심사의 비용**을 치르는 구조다. 서비스가 2~3개일 때는 과잉이지만, 이 프로젝트의 목표는 규모가 아니라 **"서비스를 하나 더 붙일 때 무엇을 복제하면 되는지가 코드로 드러나는 레퍼런스"**다. 그래서 서비스 수는 최소(인증 1 + 샘플 백엔드 1)로 유지하면서, 횡단 관심사(게이트웨이·디스커버리·SSO·컨테이너화)는 실전 수준으로 갖췄다.

repo 전략: 각 서비스는 **독립 git repo**(독립 배포 단위 = 독립 버전 이력), 우산 repo는 인프라와 조립 방법만 담는다. 설계·작업 문서는 repo가 아니라 **Obsidian에만** — [[06 저장소 구조 + 운영 원칙]].

## 2. 인증 플랫폼 — 이 시스템의 심장

### 2.1 로그인 흐름 전체 그림

```
브라우저 ─① /oauth2/authorization/keycloak─▶ auth-server
        ◀② 302 Keycloak 로그인 페이지로 ────┘
   ③ Keycloak에서 로그인 (ID/PW 또는 구글 버튼 → 구글 IdP)
        ◀④ 302 redirect_uri(/login/oauth2/code/keycloak)+인가코드
브라우저 ─⑤ 인가코드 전달─▶ auth-server ─⑥ 코드↔토큰 교환(백채널)─▶ Keycloak
                              ⑦ ID토큰 검증 → JIT 유저 생성/갱신 (authdb)
                              ⑧ 자체 RS256 JWT(AT) + RT 발급 → HttpOnly 쿠키
        ◀⑨ returnTo 쿠키의 원래 위치로 복귀
이후: 브라우저 ─(쿠키 AT)─▶ gateway ─▶ board-service (JWKS로 자체 검증)
```

> [!note] 배경지식 — OAuth2와 OIDC, 그리고 인가 코드 플로우
> **OAuth2**는 "권한 위임" 프레임워크고, **OIDC(OpenID Connect)** 는 그 위에 "인증(누구인가)"을 얹은 표준이다. 핵심인 **인가 코드 플로우(Authorization Code Flow)** 는 비밀번호가 우리 서버를 절대 거치지 않게 한다 — 사용자는 IdP(Keycloak) 화면에서만 비밀번호를 치고, 우리 서버는 1회용 인가 코드를 백채널에서 토큰으로 교환한다. 반대 개념인 **ROPC**(아이디/비번을 우리 폼으로 받아 IdP에 대신 전달)는 OAuth 2.0 Security BCP에서 사실상 폐기 권고 — 이 프로젝트의 확정 결정이기도 하다: **로그인 UI 불만은 ROPC가 아니라 Keycloak 테마로 푼다** ([[02 keycloakify 로그인 테마]]).

> [!note] 배경지식 — IdP 브로커링
> Keycloak은 자체 계정 DB이면서 동시에 **브로커**다. "구글로 로그인" 버튼을 누르면 KC가 구글의 OIDC 클라이언트가 되어 한 번 더 위임하고, 결과를 자기 세션으로 흡수한다. 우리 서비스 입장에서 IdP는 영원히 Keycloak 하나 — 구글/카카오/네이버가 늘어도 auth-server 코드는 불변이다.

### 2.2 왜 자체 JWT를 발급하나 — BFF 패턴

Keycloak도 JWT를 준다. 그런데 왜 auth-server가 **또** 발급하나?

| KC 토큰을 그대로 주면 | 자체 JWT를 발급하면 (현재 구조) |
|---|---|
| 프론트/백엔드가 KC 토큰 포맷에 종속 — KC 업그레이드·설정 변경이 전 서비스에 전파 | 토큰 계약(클레임 구조)을 **우리가 통제** — `sub`(내부 유저 id), `email`, `name`, `role`, `provider` |
| KC 세션·클라이언트 스코프가 곧 API 권한 모델 | 권한 모델을 우리 DB(User) 기준으로 — KC는 "신원 확인"만 |
| 토큰이 브라우저 JS에 노출되기 쉬움 | **HttpOnly 쿠키**로만 전달 — XSS로 토큰 탈취 불가 |

> [!note] 배경지식 — BFF(Backend For Frontend)와 토큰 은닉
> 브라우저에 액세스 토큰을 직접 쥐여주는 SPA 패턴(localStorage 저장)은 XSS 한 방에 토큰이 새는 약점이 있다. BFF 패턴은 **토큰 교환·보관을 서버(auth-server)가 대행**하고 브라우저에는 HttpOnly 쿠키만 준다. JS가 못 읽는 쿠키이므로 XSS가 나도 토큰 자체는 탈취되지 않는다(요청 위조는 별도 방어). 우리 구조는 "BFF + 자체 발급 JWT"의 조합: 쿠키 속 내용물이 KC 토큰이 아니라 **우리가 서명한 RS256 JWT**라서, 리소스 서버들이 KC를 몰라도 검증할 수 있다.

### 2.3 RS256 + JWKS — 분산 검증의 열쇠

> [!note] 배경지식 — 대칭(HS256) vs 비대칭(RS256), 그리고 JWKS
> **HS256**은 서명·검증이 같은 비밀키 — 검증자가 늘 때마다 비밀키가 복제되고, 비밀키를 가진 누구나 위조 발급이 가능하다. **RS256**은 개인키로 서명, 공개키로 검증 — 발급 능력은 auth-server에만 있고, 검증자는 공개키만 알면 된다. 공개키를 배포하는 표준 창구가 **JWKS**(JSON Web Key Set, `/.well-known/jwks.json`)다. 서비스가 10개로 늘어도 "JWKS URL 하나 설정"이 검증 배선의 전부가 된다.

우리 구현의 검증 계약 (3곳이 동일 env로 정렬 — [[07 보안 감사 + 하드닝 (2026-07-02)]]):

- `iss = ${PLATFORM_ISSUER:-http://localhost:9000}` — 발급자 확인
- `aud = ${PLATFORM_AUDIENCE:-platform-api}` — "이 API용 토큰"인지 확인 (다른 시스템용 토큰 재사용 차단)
- 서명 = auth-server JWKS 공개키
- 검증 지점: auth-server `/api/me` 디코더, board-service `JwtDecoder`, gateway 조기차단(2.6절) — **발급자 1곳, 검증은 각자**

서명키는 파일(`platform.jwk-path`, 컨테이너에선 named volume)로 영속 — 재기동해도 유통 중인 AT가 살아남는다. 키 회전(kid 다중 키)은 운영 전환 백로그.

### 2.4 토큰 수명 모델 — refresh 회전과 재사용 탐지

- **AT**: 15분, stateless — 서버가 기억하지 않는다. 검증은 순수 서명+클레임
- **RT**: DB에 **SHA256 해시만** 저장(원문 유출 대비), 사용할 때마다 **회전**(새 RT 발급, 옛 RT은 superseded 표시)
- **재사용 탐지**: superseded 처리된 옛 RT가 다시 오면 = 탈취 의심 → 그 **패밀리 전체 폐기**(강제 재로그인)

> [!note] 배경지식 — 왜 회전+패밀리 폐기인가
> RT는 수명이 길어서 탈취 시 피해가 크다. OAuth 2.0 Security BCP의 권고가 **rotation + reuse detection**: 정상 클라이언트는 항상 최신 RT만 쓰므로, "한 번 소비된 RT의 재등장"은 공격자와 피해자 중 한쪽이 낡은 토큰을 쓰고 있다는 확실한 신호다. 어느 쪽이 도둑인지 알 수 없으므로 계보(패밀리) 전체를 죽이는 것이 표준 대응. 참고로 초기 구현은 "모든 재사용 = 폐기"였다가 **superseded 토큰의 재사용만** 폐기하도록 정밀화했다 — 네트워크 재시도 같은 무해한 중복까지 죽이지 않기 위해.

### 2.5 JIT 프로비저닝과 /api/me

KC ID토큰의 사용자가 authdb에 없으면 **로그인 순간 생성**(Just-In-Time), 있으면 프로필 갱신. 별도 회원가입 화면이 없는 이유다. `/api/me`는 쿠키 AT를 검증해 `{sub, email, name, role, provider}`를 주는 프론트의 세션 확인 창구 — 3개 SPA의 AuthGate가 전부 이걸 부른다.

### 2.6 로그아웃 — 백채널 확정

로그아웃은 **서버-서버 end_session**(auth-server가 보관한 KC refresh_token 사용) + 자체 쿠키 말소. 프론트채널(id_token_hint 리다이렉트)은 id_token 만료 시 로그아웃이 실패하는 버그가 있어 **되돌리지 않기로 확정** — [[01 인증 플랫폼 (Keycloak BFF + 자체 JWT)]].

## 3. 게이트웨이 — 단일 진입점의 4가지 직무

> [!note] 배경지식 — API Gateway 패턴
> 클라이언트가 서비스 N개의 주소를 직접 알면, CORS·인증·로깅·제한 정책이 N곳으로 흩어지고 서비스 내부 구조가 외부 계약이 되어버린다. 게이트웨이는 **단일 진입점**에서 라우팅과 횡단 관심사를 처리하는 패턴. 우리는 Spring Cloud Gateway(SCG) — Netty+WebFlux 기반 **리액티브(논블로킹)** 스택이라 프록시처럼 "대기가 대부분인" 워크로드에 스레드를 적게 쓴다. 상세 코드 워크스루는 [[10 게이트웨이 동작 흐름 코드 워크스루 (2026-07-05)]].

gateway-server(:8000)의 직무 4가지:

**① 라우팅** — `/api/auth/**, /oauth2/**, /login/**` → `lb://auth-server`, `/api/board/**` → `lb://board-service`. `lb://`는 Eureka에서 인스턴스를 찾아가는 스킴(4절).

**② CORS 중앙화** — 각 서비스의 CORS 설정을 전부 **제거**하고 게이트웨이 한 곳으로. 특히 **401 응답에도 CORS 헤더**를 붙인다 — 이게 없으면 만료 AT로 API를 부른 프론트가 "401을 읽고 refresh 시도"라는 회복 흐름 자체를 못 탄다(브라우저가 응답을 가려버림).

> [!note] 배경지식 — CORS는 서버 방어가 아니라 브라우저 규칙
> CORS는 "다른 오리진의 JS가 이 응답을 읽어도 되는가"를 브라우저가 판단하게 하는 헤더 계약이다. 서버 간 호출·curl에는 아무 효력이 없다. 그래서 보안 장치가 아니라 **프론트 개발 편의의 배선**에 가깝고, 흩어질수록 디버깅 지옥이 되므로 게이트웨이 중앙화가 정석이다. (nginx 단일 오리진 모드에선 애초에 크로스 오리진이 아니라 CORS가 등장하지 않는다 — dev 모드용 배선.)

**③ 관측** — 요청 로깅 + `X-Request-Id`(traceId). 클라이언트가 보낸 값은 `[A-Za-z0-9._-]{1,64}` 검증 후 불일치 시 재발급 — **로그 인젝션 방지**(임의 문자열이 로그에 그대로 박히는 것 차단).

**④ 트래픽 제어 + 조기 차단** ([[08 게이트웨이 트래픽 제어 + JWT 조기차단 (2026-07-03)]]):

| 장치 | 설정 | 배경 |
|---|---|---|
| Rate limiting | `/api/auth/**` 5rps/버스트10, `/oauth2/** /login/**` 10rps/20, IP 키 | 인증 경로 = 크리덴셜 스터핑 표적. **토큰 버킷**(평균 rps + 순간 버스트 허용) 알고리즘, Redis가 카운터 저장소 |
| Redis fail-open | Redis 죽으면 **통과**(제한 없이) + ERROR 로그 | 가용성 우선 — 제한장치 고장이 전면 장애가 되면 안 된다. fail-closed(차단)는 로그인 전체 마비를 뜻함 |
| Circuit Breaker | board 라우트, 실패 누적 시 즉시 fallback 503 | 죽은 서비스에 계속 요청을 보내며 스레드·타임아웃을 낭비하지 않기 위한 상태기계(closed→open→half-open). Resilience4j |
| Retry | **GET + IOException 한정** | GET은 멱등이라 재시도 안전. 타임아웃까지 재시도하면 지연이 증폭되므로 연결성 오류만 |
| 전역 타임아웃 | connect 3s / response 10s, CB TimeLimiter 11s | 기본 TimeLimiter 1s가 정상 응답까지 503으로 만든 Critical을 잡은 값 — 사건 기록 참조 |
| JWT 조기차단 | 게이트웨이도 reactive resource server로 JWKS+iss/aud 검증, 공개경로 permitAll | 무효 토큰이 내부망에 들어오기 전에 401 |

> [!warning] 운영 계약 — 다층 방어의 동기화 의무
> 게이트웨이 = 1차 방어(조기 401), 각 서비스 = 최종 방어(자체 검증). 게이트웨이가 뚫려도 서비스가 막는 **defense in depth**이지만, 대신 **board-service의 공개경로가 바뀌면 gateway SecurityConfig도 같이 고쳐야 한다** — 두 곳의 공개경로 목록은 손으로 동기화하는 계약이다.

## 4. 유레카 — 서비스 디스커버리

> [!note] 배경지식 — 왜 주소를 하드코딩하지 않나
> 게이트웨이 설정에 `http://localhost:9000`을 박으면, 포트 변경·다중 인스턴스·컨테이너 전환 때마다 게이트웨이를 고쳐야 한다. **클라이언트 사이드 디스커버리**는 각 서비스가 기동 시 레지스트리(Eureka)에 자기 주소를 등록하고 하트비트로 생존 신고, 호출자는 "이름"으로 조회해 직접 호출하는 방식이다. `lb://auth-server`의 `auth-server`가 바로 Eureka 등록명. 인스턴스가 늘면 게이트웨이가 자동으로 로드밸런싱한다 — [[09 유레카 서비스 디스커버리 (2026-07-05)]].

- eureka-server(:8761), 대시보드에서 등록 현황 확인
- `prefer-ip-address: true` — Windows 호스트명 해석 문제 회피(로컬 dev 사정). 컨테이너 네트워크에선 서비스명 DNS 재검토가 이월 과제
- 기동 순서 자유 — 등록되는 대로 라우팅이 살아난다

## 5. 데이터 층

- **PostgreSQL 1대(:5433), DB 3개**: `keycloak`(KC 전용) / `authdb`(User, RefreshToken) / `boarddb`(게시글·댓글). `init-authdb.sql`이 최초 볼륨 초기화 때 authdb·boarddb를 생성 — `down -v` 후 board 부팅 실패를 잡은 수정이 여기 있다
- **Flyway** — 스키마 마이그레이션을 코드로 버전 관리. Boot 4에서 `spring-boot-flyway` 모듈 누락으로 부팅 실패했던 사건이 "의존성까지가 마이그레이션 설정"임을 남겼다
- **Redis(:6379)** — 게이트웨이 rate limit 카운터 전용, 선택적(없으면 fail-open)

> [!note] 배경지식 — "DB는 서비스마다"의 현실적 타협
> MSA 순수주의는 서비스별 DB **서버**를 요구하지만, 로컬/소규모에선 한 서버에 **논리 DB 분리**로 같은 효과(스키마 격리, 계정 분리 가능)를 얻는 것이 관례다. 서비스가 서로의 테이블을 직접 조회하지 않는다는 규율이 본질이고, 물리 분리는 부하가 생길 때 해도 늦지 않다.

- 배포: tsup/Vite lib 빌드 + Changesets. npmjs 배포는 보류(2FA 마찰) — **tarball을 소비 앱이 직접 설치**하는 방식으로 검증 완료. alm/wiki가 실제 소비자

> [!note] 배경지식 — 디자인 토큰과 headless UI
> 디자인 토큰은 "색·간격·타이포를 이름 붙은 변수로 승격"하는 것 — 값이 아니라 의미(`color-border-danger`)로 참조하므로 테마 전환(다크모드)이 변수 재정의 하나로 끝난다. Radix 같은 headless UI는 **동작과 접근성(키보드, ARIA, 포커스 트랩)만 제공하고 스타일은 백지** — "보이는 것"은 토큰으로 우리가 결정하고, "움직이는 것"의 난제(포커스 관리, roving tabindex)는 검증된 라이브러리에 맡기는 분업이다. 웨이브 기록의 사건들(Radix setTimeout 포커스 경쟁, Tooltip 이중 노드, Toast 아나운서)이 전부 이 경계에서 나왔다.

### 6.3 SSO 게이트 — 앱 3개가 로그인을 공유하는 방법

각 앱에 **AuthGate**: 마운트 시 `/api/me` 호출 → 200이면 사용자 표시, 401이면 refresh 시도 후 로그인으로. 로그인 시 **returnTo 쿠키**에 원래 위치를 담아 로그인 완료 후 그 자리로 복귀. 단일 오리진(7절)이라 쿠키가 세 앱에 자동 공유 — **앱 A에서 로그인하면 B·C는 이미 로그인 상태**다.

## 7. nginx 단일 오리진 — SSO가 성립하는 물리적 이유

> [!note] 배경지식 — 쿠키는 오리진(호스트) 단위다
> 도메인 지정 없이 발급된 쿠키(host-only)는 **정확히 같은 호스트**로만 전송된다. dev 모드처럼 :5173/:5174/:5175로 나뉘면… 사실 쿠키는 포트를 구분하지 않아 전송은 되지만, 오리진이 달라 CORS·리다이렉트 처리가 전부 크로스 오리진 배선이 된다. 반면 세 앱을 `http://localhost` 하나의 **경로**(`/`, `/wiki/`, `/alm/`)로 합치면 쿠키·오리진 문제가 통째로 소멸한다. **nginx 1대 = 절약이 아니라 SSO 설계 그 자체** — [[11 nginx 통합배포 + 3앱 SSO (2026-07-14)]], [[12 크로스앱 내비게이션 + redirect_uri 사건 + 운영 전환 로드맵 (2026-07-15)]].

`infra/nginx/default.conf`의 결정들:

- 정규식 location `~ ^/(api|oauth2|login|\.well-known)/` → gateway 프록시. **`/login`(슬래시 없음)은 매칭 안 됨** → myFront의 SPA 로그인 페이지가 산다 — 정규식 끝 `/` 하나가 SPA 라우트와 API 경로를 가르는 스위치
- SPA 서빙은 `root` + `try_files $uri /<app>/index.html` — 딥링크 새로고침을 index.html 폴백으로. alias+try_files 조합의 고전 버그를 피해 root 방식
- `X-Forwarded-Host/Proto` 부착, **X-Forwarded-Port는 의도적으로 안 보냄** — `$server_port`는 컨테이너 내부 포트(80) 고정이라 `NGINX_PORT≠80`일 때 오히려 거짓말이 된다

> [!note] 배경지식 — X-Forwarded-* 체인과 trusted proxies
> 프록시 뒤의 서버는 "브라우저가 원래 두드린 주소"를 모른다. 그걸 전달하는 관례가 `X-Forwarded-Host/Proto/For`이고, auth-server는 이걸로 OIDC `redirect_uri`를 조립한다(`forward-headers-strategy: framework`). 문제는 이 헤더가 **위조 가능한 평문**이라는 것 — 그래서 SCG 4.1+는 `trusted-proxies` 미설정 시 헤더를 **버린다**. 이 보안 변경을 몰라서 터진 것이 redirect_uri 사건의 원인 ①이었다([[12 크로스앱 내비게이션 + redirect_uri 사건 + 운영 전환 로드맵 (2026-07-15)|12]]). 현재는 루프백+사설대역을 신뢰(dev 전용), 운영에선 실제 LB 대역으로 좁히는 것이 계약.

## 9. 교차 계약 모음 — 이것만은 어기면 안 된다

여러 repo에 걸쳐 있어 한 곳만 고치면 깨지는 계약들:

| 계약 | 걸린 곳 | 내용 |
|---|---|---|
| `PLATFORM_ISSUER` / `PLATFORM_AUDIENCE` | auth 발급 ↔ auth /api/me ↔ board ↔ gateway | 같은 env 이름·기본값(`http://localhost:9000` / `platform-api`) 3~4곳 동일 |
| JWKS URI | board·gateway의 `AUTH_JWKS_URI` | auth-server의 `/.well-known/jwks.json` |
| 공개경로 목록 | board SecurityConfig ↔ gateway SecurityConfig | 한쪽 변경 시 **수동 동기화** (3절 경고) |
| `FRONTEND_URL` | auth-server | dev=`http://localhost:5173`, nginx 모드=`http://localhost` — IntelliJ에 Run Config 2개인 이유 |
| KC 프론트/백채널 | auth docker 프로필 | 브라우저=`localhost:8080`, 백채널=`keycloak:8080`, iss 검증=`localhost:8080`(`KEYCLOAK_FRONT_ISSUER`) |
| realm 변경 | realm-export.json ↔ 실행 중 KC DB | `--import-realm`은 기존 realm을 안 덮는다 — **json 수정 + kcadm 반영이 한 세트** |
| Vite base | wiki=`/wiki/`, alm=`/alm/` | nginx location과 1:1. 크로스앱 링크는 통합 오리진에서만 동작 |
| POSTGRES_PASSWORD | compose ↔ auth ↔ board | `.env` 한 곳에서 주입, 기본값은 dev 전용 |
