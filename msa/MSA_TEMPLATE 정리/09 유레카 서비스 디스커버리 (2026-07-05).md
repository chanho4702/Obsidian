# 9단계 — 유레카 서비스 디스커버리 (lb:// 전환)

상위: [[00 개요 — 전체 구조]] · 선행: [[08 게이트웨이 트래픽 제어 + JWT 조기차단 (2026-07-03)]] (남은 확장 포인트 1번 이행)

"게이트웨이 라우트가 전부 고정 URI 환경변수"라는 한계에서 출발 — 주소 변경·스케일아웃 때마다
게이트웨이 재기동이 필요한 구조를 동적 디스커버리로. 스펙: `specs/2026-07-05-eureka-service-discovery-design.md`

## 핵심 개념 — 서비스 디스커버리가 푸는 문제

게이트웨이는 요청을 뒤로 넘기려면 "board-service가 지금 어디(IP:포트)에 있나"를 알아야 한다.

| 방식 | 동작 | 한계/장점 |
|---|---|---|
| **정적 설정** (기존) | URI를 환경변수에 박음 | 주소 변경 = 설정 수정 + 재기동. 다중 인스턴스 분산 불가 |
| **동적 디스커버리** (유레카) | 서비스가 기동 시 자기 등록, 게이트웨이는 이름으로 조회 | 주소·대수가 바뀌어도 게이트웨이 무변경 |

유레카 3요소:
1. **Eureka Server** — 레지스트리. "누가 어디에 살아있나"의 단일 장부 (대시보드 :8761)
2. **자기 등록(self-registration)** — 각 서비스가 기동하며 `spring.application.name`으로 등록,
   이후 **하트비트**(기본 30s)로 생존 신고. 하트비트 끊기면 레지스트리에서 **퇴출(eviction)**
3. **클라이언트 조회** — 게이트웨이가 레지스트리를 받아와 `lb://board-service`를 실제 주소로 해석

## 개념 — lb:// 와 클라이언트 사이드 로드밸런싱

```yaml
- id: board
  uri: lb://board-service   # ← spring.application.name과 일치해야 함
```

- `lb://` 스킴은 **Spring Cloud LoadBalancer**가 처리 (eureka-client 의존성에 전이 포함)
- "board-service 인스턴스 목록을 유레카에서 받아 그중 하나로 보내라" — 기본 라운드로빈
- **클라이언트 사이드**인 이유: 분산 결정을 호출하는 쪽(게이트웨이)이 함 — 중간에 nginx/L4 없이
  스케일아웃. (↔ 서버 사이드 LB: 별도 장비가 분산)

## 개념 — 유레카는 SPOF가 아니다 (로컬 캐시)

유레카 클라이언트는 레지스트리 스냅샷을 **로컬 캐시**하고 30초마다 갱신한다.
→ 유레카 서버가 죽어도 마지막 스냅샷으로 라우팅은 계속됨. 유레카 장애 = "변경 반영이 멈춤"이지
"라우팅 중단"이 아니다. (같은 이유로 반영 지연도 최대 수십 초 존재 — 즉시성 아님)

## 개념 — 기존 resilience 체계와의 역할 분담 (상호보완)

| 계층 | 역할 | 판단 근거 |
|---|---|---|
| 유레카 | 죽은 인스턴스를 **목록에서 제거** | 하트비트 (프로세스 생존) |
| CircuitBreaker ([[08 게이트웨이 트래픽 제어 + JWT 조기차단 (2026-07-03)]]) | 목록엔 있는데 **응답이 이상한** 곳 차단 | 실제 응답 실패율 |

미등록 시점 요청은 lb:// 해석 실패 503 → board는 기존 `/fallback/board`가 받음. 기동 순서 강제 없음.

## 설계 결정 (와 그 이유)

1. **eureka-server는 독립 Gradle 프로젝트** — auth/board와 같은 패턴. 템플릿 학습용으로 코드가 보이게
2. **단일 노드, HA 없음 (YAGNI)** — `register-with-eureka: false` + `fetch-registry: false`
   (standalone: 자길 등록할 레지스트리도, 조회할 peer도 없음)
3. **라우트 6개 전부 lb:// 전환** — board만 하고 auth는 고정 URI로 두면 두 방식 혼재. 일관성 우선
4. **`platform.jwks-uri`는 직접 URI 유지** — 게이트웨이 내부 JwtDecoder의 HTTP 호출은
   라우팅을 안 타므로 lb://가 적용 안 됨. load-balanced WebClient 도입은 오버엔지니어링
5. **K8s로 가면 유레카는 중복** — K8s Service DNS가 디스커버리+LB를 대체 (`http://board-service`).
   유레카가 진가를 발휘하는 건 VM/온프레/compose 환경에서 인스턴스가 동적으로 늘고 줄 때

## E2E에서 잡힌 실동작 결함 (교훈 — 유닛테스트는 전부 green이었다)

- **E1 (Critical)**: Windows/Hyper-V 환경에서 서비스가 유레카에 `DESKTOP-xxx.mshome.net`
  호스트명으로 등록됨 → 게이트웨이 Netty DNS가 NXDOMAIN → **모든 lb:// 라우팅이
  UnknownHostException 500**. 세 서비스 모두 `eureka.instance.prefer-ip-address: true`로
  IP 등록 강제하여 해결. **유레카 도입 시 prefer-ip-address는 필수 체크리스트**
  (컨테이너/VM/Windows 어디서든 호스트명 해석은 신뢰 불가).
- **E2**: Docker Desktop(WinNAT)이 부팅 시 예약한 포트 제외 범위(9008–9107)에 :9100이
  걸려 board-service가 "port already in use"(리스너는 없음)로 기동 실패.
  `netsh interface ipv4 show excludedportrange protocol=tcp`로 확인, 범위 밖 포트로 우회
  (최종 :9710 — :9700도 다른 프로세스의 아웃바운드 임시 포트에 우연히 점유당했었음.
  `Get-NetTCPConnection -LocalPort <p>`로 확인) —
  **유레카 덕에 포트를 몇 번을 바꿔도 게이트웨이 설정 무변경**이라는 도입 가치가 즉석에서 증명됨.

## E2E 검증 결과 (전부 통과)

1. 대시보드(:8761)에 GATEWAY-SERVER·AUTH-SERVER·BOARD-SERVICE 3개 UP (IP 등록 확인)
2. 게이트웨이 경유 `GET /api/board/posts`(lb://board-service)·`GET /.well-known/jwks.json`(lb://auth-server) → 200
3. board 강제 종료 → 하트비트 퇴출 + 캐시 갱신 후 `503 {"error":"board_unavailable"}` 수렴 (기존 CB fallback과 합성 확인)
