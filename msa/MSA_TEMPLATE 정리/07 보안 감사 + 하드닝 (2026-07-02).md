# 7단계 — 전체 보안 감사 + 하드닝

상위: [[00 개요 — 전체 구조]] · 관련: [[01 인증 플랫폼 (Keycloak BFF + 자체 JWT)]], [[05 API 게이트웨이 설계 (미래)]]

게이트웨이 도입까지 끝난 상태에서 5개 컴포넌트(myFront / gateway-server / auth-server / board-service / infra) 전체를 보안 관점으로 감사하고, 발견된 약점을 TDD로 강화했다. 독립 리뷰어 검증 결과 **READY (Critical 0)**.

## 감사에서 "이미 잘 돼 있던 것" (변경 안 함)

- 프론트: AT는 JS 메모리에만, RT는 HttpOnly 쿠키 — localStorage/sessionStorage에 토큰 없음.
- RT 회전 + 재사용 탐지(도난 시 패밀리 전체 폐기), DB에는 SHA-256 해시만 저장.
- Keycloak: ROPC(직접 패스워드 그랜트) 비활성, redirect URI 와일드카드 없이 정확 한정.
- JWKS 엔드포인트는 공개키만 노출.
- 시크릿 파일 커밋 여부: `auth-server/auth-jwk.json`(RSA 개인키), `infra/keycloak/.env`(구글 시크릿) 둘 다 **gitignore 확인, git 미추적** — 커밋 유출 없음.

## 적용된 강화 (커밋별)

### auth-server `1081692` — feat(security)
- **AT에 `aud=platform-api` 클레임 추가** (JwtService). 이 토큰이 "이 플랫폼 API용"임을 명시.
- **/api/me 디코더가 서명 외에 issuer/audience까지 검증** (SecurityConfig — `JwtValidators.createDefaultWithIssuer` + `JwtClaimValidator(aud)`). 다른 발급자·다른 용도 토큰 거부.
- **CookieFactory `secure(false)` 하드코딩 제거** → `platform.cookie-secure`(env `COOKIE_SECURE`, dev 기본 false). 운영 https에서 true 주입.
  - SameSite=Lax는 프론트·게이트웨이 same-site 전제 — cross-site 배포로 바뀌면 None+Secure 필요(코드 주석에 기록).
- **application.yml 시크릿 전부 env 외부화**: `AUTH_DB_URL/USERNAME/PASSWORD`, `OIDC_CLIENT_ID/SECRET`, `KEYCLOAK_ISSUER_URI`, `PLATFORM_ISSUER/AUDIENCE`, `FRONTEND_URL`, TTL 2종. 기본값 = 로컬 dev 값이라 dev 흐름 불변.
- 테스트: `JwtDecoderValidationTest`(정상/타issuer/타aud — 실제 키로 서명한 진짜 토큰으로 검증), `CookieFactoryTest` 신규(기존 follow-up 해소), logout `Max-Age=0`·me `name/provider` 어서션 보강. **전체 green.**

### board-service `111db1e` — feat(security)
- **JwtDecoder 빈 신설**: 기존엔 JWKS **서명만** 검증 → issuer + `AudienceValidator`(aud=platform-api) 검증 추가. 서명이 유효해도 발급자/대상이 다르면 401.
- **AccessGuard**: 비숫자 sub 토큰이 `NumberFormatException`(500)을 내던 것 → `AccessDeniedException`(403).
- yml env 외부화: `BOARD_DB_*`, `AUTH_JWKS_URI`, `platform.jwt.issuer/audience`(`PLATFORM_ISSUER/AUDIENCE`).
- 테스트: `AudienceValidatorTest` 3케이스 + AccessGuard 비숫자 sub 케이스. **전체 green.**

### gateway-server `e218e1b` — feat(security)
- **X-Request-Id 검증**: 클라이언트가 준 값을 무검증으로 로그·다운스트림에 싣던 것(로그 인젝션/헤더 오염 표면) → `[A-Za-z0-9._-]{1,64}` 전체 매칭 검증, 불일치 시 UUID 재발급.
- CORS `maxAge: 3600`(프리플라이트 캐시). credentials+구체 오리진 조합 유지 — **`CORS_ALLOWED_ORIGIN`에 `*` 넣으면 안 됨**(yml 주석으로 고정).
- 테스트: 특수문자/과길이 재발급 2케이스. **전체 green.**

### umbrella(infra) `215c7bc` — chore(infra)
- **realm-export.json**: `sslRequired: none → external`(localhost·사설IP는 http 계속 허용), **brute-force 방어 활성화**(5회 실패 → 60s~최대 900s 잠금), 토큰·세션 수명 명시(`accessTokenLifespan 300` / `ssoSessionIdle 1800` / `ssoSessionMax 36000` — KC 기본값과 동일, 명시화).
- **docker-compose**: `POSTGRES_PASSWORD`·`KC_ADMIN_USERNAME/PASSWORD` env 파라미터화(기본=dev값), postgres healthcheck(`pg_isready -h 127.0.0.1`, `start_period 10s`) + `depends_on: service_healthy`.
- **.env.example**: 비밀번호 바꿀 때 auth/board 서비스 env(`AUTH_DB_PASSWORD`/`BOARD_DB_PASSWORD`)도 같이 줘야 한다는 교차 계약 + "env는 최초 볼륨 초기화 때만 적용" 주의 문서화.

## 반영 주의 / 남은 수동분

- **realm 변경 반영**: `cd infra/keycloak && docker compose down -v && docker compose up -d` (재import — 기존 게이트웨이 세션의 수동분과 동일 절차, DB 볼륨 초기화됨).
- 기동 중이던 aud 없는 구 AT는 401 → 프론트 refresh 흐름이 새 AT를 받아 자연 회복(최대 15분 창).
- brute-force가 켜져서 **실패 로그인 반복 테스트 시 60초 잠금** 걸릴 수 있음(alice 5회 오타 등).
- auth-server·gateway-server README에 이전 세션 문서보강 diff가 **미커밋으로 남아 있음**(이번 작업과 무관, 보존).
- push는 여전히 전부 보류 상태(로컬 커밋만).
