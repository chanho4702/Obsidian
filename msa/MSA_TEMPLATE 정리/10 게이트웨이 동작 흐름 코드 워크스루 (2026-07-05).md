# 10 — 게이트웨이 동작 흐름 코드 워크스루

상위: [[00 개요 — 전체 구조]] · 선행: [[08 게이트웨이 트래픽 제어 + JWT 조기차단 (2026-07-03)]](이 노트는 그 결과물을 요청 흐름 순서로 다시 읽는 학습 기록)

gateway-server(:8000)의 코드를 **요청 하나가 지나가는 순서대로** 분석한 노트. 파일별 정리가 아니라 실행 순서별 정리라는 점이 08과의 차이.

## 큰 그림: 두 층으로 나뉜 파이프라인

Spring Cloud Gateway는 WebFlux(Netty) 위에서 돌고, 처리 순서가 두 층이다:

```
클라이언트 요청 (:8000)
   │
   ▼
[1층: WebFilter 체인 — 라우팅 "이전"]
   ├─ ① CORS 처리 (Security 안에 통합)
   ├─ ② Spring Security: 경로 인가 + JWT 검증  ← 여기서 걸리면 401, 게이트웨이 필터까지 못 감
   ▼
[2층: Gateway 라우팅 — DispatcherHandler]
   ├─ ③ RoutePredicateHandlerMapping: Path 프레디케이트로 라우트 매칭
   ├─ ④ RequestLoggingFilter (GlobalFilter, 게이트웨이 필터 중 최우선)
   ├─ ⑤ 라우트별 필터: RequestRateLimiter 또는 CircuitBreaker→Retry
   ├─ ⑥ NettyRoutingFilter: 실제 다운스트림 HTTP 호출 (connect 3s / response 10s)
   ▼
auth-server(:9000) 또는 board-service(:9100)
```

> [!important] 흔한 오해
> `RequestLoggingFilter`의 `HIGHEST_PRECEDENCE`는 **게이트웨이 필터들 중** 1등이라는 뜻. Security는 WebFilter라서 그보다 앞이다. 따라서 **Security가 401로 쳐낸 요청은 로깅 필터에 도달조차 못 한다** — 401 요청이 액세스 로그에 안 남는 이유.

## ① CORS — `SecurityConfig.corsConfigurationSource()`

```java
config.setAllowedOrigins(List.of(allowedOrigin));  // http://localhost:5173 (구체 값)
config.setAllowCredentials(true);
config.setMaxAge(3600L);
```

- `allowCredentials(true)`는 쿠키(RT)를 실어 보내기 위한 것 — 이게 켜지면 CORS 스펙상 `Allow-Origin: *` **금지**. 그래서 오리진은 env로 구체 값 하나.
- `maxAge(3600)` — preflight 결과를 브라우저가 1시간 캐시, OPTIONS 왕복 절약.
- yml `globalcors` 대신 **Security의 `.cors(...)`에 꽂은 이유**: CORS 헤더가 Security 레벨에서 붙어야 **401 응답에도** 붙는다. 없으면 "401을 읽고 → refresh" 하는 프론트 흐름이 브라우저 CORS 에러로 막힌다(401인데 CORS 에러로 보여 디버깅 지옥).

## ② 인가 + JWT 검증 — `securityWebFilterChain()`

```java
.pathMatchers(HttpMethod.OPTIONS).permitAll()                    // preflight 무조건 통과
.pathMatchers("/oauth2/**", "/login/**", "/api/auth/**",
        "/.well-known/**", "/fallback/**").permitAll()           // 인증 "받으러 가는" 경로
.pathMatchers(HttpMethod.GET, "/api/board/posts/**").permitAll() // 비로그인 게시글 열람
.anyExchange().authenticated()                                   // 나머지 전부 토큰 필수
```

- `/oauth2` `/login` `/api/auth`는 토큰을 **받으러 가는** 경로라 토큰 없이 열려야 함. `/.well-known`은 공개키 배포 경로.
- `/fallback/**` permitAll — 서킷브레이커가 fallback으로 forward한 요청이 Security에 걸려 401로 죽는 것을 막는 방어.
- `csrf/httpBasic/formLogin` disable — 게이트웨이는 세션 없는 순수 토큰 검증기.
- `.oauth2ResourceServer(jwt)`가 실제 검증 스위치: `Authorization: Bearer` 추출(없으면 즉시 401) → 디코더 검증 → 실패 401 / 성공 통과.

## ③ JWT 디코더 — `reactiveJwtDecoder()`

```java
NimbusReactiveJwtDecoder decoder = NimbusReactiveJwtDecoder.withJwkSetUri(jwksUri).build();
decoder.setJwtValidator(new DelegatingOAuth2TokenValidator<Jwt>(
        JwtValidators.createDefaultWithIssuer(issuer),
        new AudienceValidator(audience)));
```

검증은 **서명 → 클레임** 2단계:

1. **서명**: auth-server `/.well-known/jwks.json`의 공개키(내부 캐시)로 "진짜 auth-server 발급인가" 확인. 위조는 여기서 탈락.
2. **클레임** (Delegating 체인): `exp`(만료) + `iss`(발급자) + `aud`(아래 ④).

## ④ `AudienceValidator` — 토큰 혼용 방지

```java
if (jwt.getAudience() != null && jwt.getAudience().contains(audience)) return success();
```

**서명이 유효한 것 ≠ 이 API용 토큰인 것.** 같은 IdP가 서비스 A용/B용 토큰을 다 발급하면, 서명 검증만으로는 A용 토큰으로 B를 호출하는 걸 못 막는다(confused deputy). `aud` 클레임이 그 구분자. `contains`인 이유는 `aud`가 스펙상 배열일 수 있어서.

## ⑤ 라우트 매칭 — application.yml

| 라우트 | Path | 목적지 | 필터 |
|---|---|---|---|
| `board` | `/api/board/**` | :9100 | CircuitBreaker + Retry |
| `auth-oauth2` | `/oauth2/**` | :9000 | RateLimiter 10/s burst 20 |
| `auth-login` | `/login/**` | :9000 | RateLimiter 10/s burst 20 |
| `auth-api` | `/api/auth/**` | :9000 | RateLimiter 5/s burst 10 |
| `auth-jwks` | `/.well-known/**` | :9000 | 없음 |
| `auth-me` | `/api/me` | :9000 | 없음(의도) |

`auth-me`에 rate limiter가 없는 이유: `authenticated()` 경로라 무토큰/위조는 Security가 이미 401로 쳐냈고 **유효 토큰 소지자만 도달** → 브루트포스 표면이 아님. 반면 `/api/auth/**`(refresh/logout)는 permitAll이라 누구나 두드릴 수 있어 가장 빡빡(5/s).

## ⑥ `RequestLoggingFilter` — 분산 추적의 시작점

```java
String requestId = exchange.getRequest().getHeaders().getFirst(REQUEST_ID);
if (requestId == null || !SAFE_ID.matcher(requestId).matches()) {
    requestId = UUID.randomUUID().toString();
}
```

- 클라이언트 `X-Request-Id`가 `[A-Za-z0-9._-]{1,64}` 형식이면 **유지**(프론트 발급 ID로 끝까지 추적), 아니면 UUID 재발급. 검증 없이 실으면 개행문자로 가짜 로그 줄을 만드는 **로그 인젝션** 가능.
- `exchange.mutate()` — WebFlux 요청 객체는 불변이라 복제본 필요.
- `h.set()` (add 아님) — add면 클라이언트 값+우리 값 헤더 **2개**가 내려가 다운스트림이 모호해짐. set은 항상 1개.
- 이 ID 하나로 게이트웨이→auth→board 3개 서비스 로그를 꿴다.

## ⑦-a auth 경로: RequestRateLimiter + `ipKeyResolver`

- `ipKeyResolver` 빈이 버킷 키(클라이언트 IP)를 결정. yml에서 `key-resolver: "#{@ipKeyResolver}"` SpEL 참조(빈 이름=메서드명).
- 동작은 Redis **토큰 버킷**: IP마다 `replenishRate`(초당 충전) / `burstCapacity`(최대치). 예: auth-api 5/s + burst 10 → 순간 10발, 지속 초당 5발. 초과 시 **429** 즉시 반환(다운스트림에 안 감).
- Redis 죽으면 **fail-open**(무제한 통과) — "Redis 장애 → 로그인 전면 장애" 전파보다 "잠시 제한 해제"를 선택한 트레이드오프.
- 주의: 이 게이트웨이 앞에 프록시/LB를 두면 `getRemoteAddress()`가 전부 프록시 IP → **전 사용자가 버킷 하나 공유**. 그땐 `X-Forwarded-For` 필요.

## ⑦-b board 경로: CircuitBreaker(바깥) → Retry(안쪽)

yml 선언 순서 = 감싸는 순서:

```
CircuitBreaker [ Retry [ 실제 프록시 호출 ] ]
```

**Retry (안쪽)**
- `methods: GET` — 멱등 GET만. POST 재시도는 "게시글 2개 생성" 사고.
- `exceptions: java.io.IOException` — **연결 실패만** 재시도. 기본값엔 TimeoutException이 포함돼 10s 타임아웃×3회=30s 지연 증폭이 생김([[08 게이트웨이 트래픽 제어 + JWT 조기차단 (2026-07-03)|08의 I1]]). 연결 거부는 ms 단위 실패라 재시도가 싸다.

**CircuitBreaker (바깥)**
- 전체 시도(재시도 포함)의 성공/실패를 기록. 실패율 임계 초과 시 **OPEN** → 이후 요청은 board를 건드리지도 않고 즉시 fallback. 죽어가는 서비스에 트래픽을 계속 부어 회복을 방해하는 것을 방지.
- `fallbackUri: forward:/fallback/board` — 외부가 아니라 게이트웨이 **내부 컨트롤러로 forward**.

**TimeLimiter 11s** — CB에는 "한 시도의 총 예산" TimeLimiter가 붙는데 기본 **1초**. 방치하면 1.2초 걸린 **정상 응답**도 fallback 503으로 잘린다([[08 게이트웨이 트래픽 제어 + JWT 조기차단 (2026-07-03)|08의 C1]]). httpclient response-timeout(10s)보다 크게 11s → 타임아웃 판정은 httpclient가, TimeLimiter는 최후 안전망.

## ⑧ `FallbackController`

```java
@RequestMapping("/fallback/board")  // → 503 {"error":"board_unavailable"}
```

클라이언트는 "connection refused로 한참 기다리다 끊김" 대신 **즉시 깔끔한 503 JSON**을 받는다. 프론트는 `board_unavailable`을 보고 "게시판 점검 중" UI를 띄우면 된다.

## 시나리오로 꿰어보기

**A. 비로그인 게시글 조회 — `GET /api/board/posts/1`**
CORS → Security(GET permitAll, 토큰 검사 없음) → `board` 라우트 → 로깅 필터 ID 발급 → CB(닫힘) → Retry 층 통과 → :9100 프록시 → 응답.

**B. 로그인 글 작성 — `POST /api/board/posts` + Bearer**
Security `authenticated()` → 토큰 추출 → JWKS 서명 → exp/iss/aud → 통과 → A와 동일(단 POST라 Retry 절대 재시도 안 함).

**C. 만료 토큰으로 B 시도**
Security exp 검증 탈락 → **401 즉시 반환**(로깅·라우팅 도달 안 함). CORS가 Security 레벨이라 이 401에도 헤더가 붙음 → 프론트가 401 읽고 `POST /api/auth/refresh`(permitAll + 5/s) → 새 토큰으로 재시도.

**D. board-service 사망 시 `GET /api/board/posts/1`**
프록시 → connection refused(IOException) → Retry 2회 추가 시도(GET+IOException 충족) → 전부 실패 → CB 실패 기록 + forward → 503. 실패가 쌓이면 서킷 OPEN → 이후 요청은 시도 없이 즉시 503, board는 회복할 숨통.

## 한 줄 요약

**Security가 문지기(누구냐), 라우트+필터가 교통정리(어디로, 얼마나 빨리), CB/fallback이 보험(뒤가 무너져도 앞은 안 무너지게)** — 세 층이 각자 한 가지씩만 담당한다.
