# 15 — ALM·Wiki 백엔드 요구사항 + 서비스 분할 설계 (2026-07-19)

상위: [[00 개요 — 전체 구조]] · 선행: [[13 백엔드 컨테이너화 + compose 통합 — CI-CD Wave 1 (2026-07-15)]], [[14 전체 구현 심화 해설 — 배경지식으로 읽는 현재 아키텍처 (2026-07-17)]]

## 무엇을 하려는가

MSA_TEMPLATE 위에 **ALM + Wiki 서비스를 온프렘 설치·운영 가능한 제품 형태**로 올리기 위한 요구사항 정리와 백엔드 서비스 분할 설계. 이 문서는 구현기록이 아니라 **설계 확정 전 논의 문서**다 — 확정된 결정과 미확정 질문을 구분해 적는다.

## 1. 요구사항 (사용자 진술 기반)

- 백엔드를 **MSA 구조 멀티 프로젝트 형태**로 구현
- 내부 서비스 간 통신은 **gRPC / REST / GraphQL을 적재적소에 혼용**
- 인프라는 필요에 따라 **MySQL / PostgreSQL / Kafka 등** 자유롭게 채택
- **성능이 중요하고 검색 속도가 좋아야** 함 → **Elasticsearch 도입 계획**
- DB가 바쁘면 **read/write 분리** 등 부하 분산 방안 고려
- 최종 목표: **ALM·Wiki 서비스를 온프렘(on-premise) 설치·운영**하기 위한 최적 설계

### 확정된 결정 (2026-07-19 대화)

| 항목 | 결정 |
|---|---|
| 1차 구현 범위 | **Wiki(문서·버전·검색) + 이슈 트래커 + 프로젝트·조직 관리 + 알림·활동 스트림** 전부 |
| 도메인 분할 | **wiki-backend / alm-backend를 별도 도메인 단위로 분리** |
| 서비스 분할안 | 아래 **B안(도메인 2 + 횡단 3)** 채택 방향 |
| 규모 목표 | **M(중소 조직 ~500명) 기준 설계 + S 축소 설치 가능**. L 전용 요소는 확장 경로만 명시 |
| 기반 스택 | Spring Boot + Spring Cloud Gateway + Keycloak **유지**. Eureka는 **(c) 프로필 분기** — 개발 모드(IDE)는 Eureka, 배포판(docker 프로필)은 컨테이너 DNS 고정. 장기적으로 배포판은 CoreDNS/k8s 디스커버리로 수렴, Eureka는 개발 편의로만 |
| repo 구조 | **도메인 2 + 플랫폼 1** — wiki-backend·alm-backend는 각각 독립 repo(제품 경계), 횡단 3서비스(org/search/notification)는 platform-backend 멀티모듈 repo 하나. common-proto 모듈(gRPC proto + Kafka 이벤트 스키마)은 platform-backend에서 GitHub Packages로 발행, wiki·alm은 의존성으로 소비. 신규 repo 3개·CI 3개 |
| DB | **PostgreSQL 통일** (MySQL 혼용 안 함) — 온프렘 설치판 DB 엔진 1종(백업·업그레이드·안내 단일화), 이슈 커스텀 필드 jsonb+GIN, S 티어 ES 폴백용 FTS. 기존 Postgres 인스턴스에 서비스별 논리 DB(wiki_db·alm_db·org_db·notif_db) + 계정 분리로 경계 유지. 특정 서비스 부하 폭증 시 그 서비스만 국소 전환 가능 |

> [!note] Eureka 프로필 분기의 근거
> M 규모 온프렘 배포는 compose/단일노드 k8s라 서비스 주소가 컨테이너 DNS로 고정되고 레플리카도 설치 시점에 정해진다 — 동적 디스커버리의 존재 가치가 없고 설치판 컨테이너·기동순서·하트비트 설명 포인트만 늘어난다. L로 가도 k8s 디스커버리를 쓰지 Eureka를 쓰진 않는다. 다만 학습 자산 보존 + IDE 개발 모드 편의를 위해 auth-server `docker` 프로필 분기 패턴([[13 백엔드 컨테이너화 + compose 통합 — CI-CD Wave 1 (2026-07-15)]])과 같은 방식으로 두 경로를 유지한다.

## 2. 서비스 분할 — 검토한 3안

분할 기준으로 쓴 원칙 세 가지. 이 질문에 "예"인 경계에서만 자른다:

1. **데이터 소유권**이 명확히 갈리는가 (위키 문서 vs 이슈)
2. **부하 패턴**이 다른가 (쓰기형 vs 검색형 vs 이벤트소비형)
3. 따로 **스케일하거나 생략**할 일이 있는가 (온프렘 고객별 옵션 구성)

### A안 — 대형 2분할 (wiki-backend / alm-backend만)

알림·검색·권한을 각 서비스에 내장. 배포 단위 최소·운영 최단순이지만, 검색 색인·알림 로직이 **양쪽에 중복 구현**되고 "통합 검색"(위키+이슈 동시 검색)·"통합 알림함"을 만들 때 결국 다시 쪼개게 된다.

### B안 — 도메인 2 + 횡단 3 ★ 채택 방향

| 서비스 | 책임 | 저장소 | 인터페이스 |
|---|---|---|---|
| **wiki-service** | 스페이스·페이지·버전·첨부 | PostgreSQL | REST(외부) / gRPC(내부) / Kafka 발행 |
| **alm-service** | 프로젝트·이슈·워크플로·라벨·댓글 | PostgreSQL (또는 MySQL) | REST / gRPC / Kafka 발행 |
| **org-service** | 조직·팀·멤버·RBAC(스페이스/프로젝트 권한) | PostgreSQL | gRPC 제공(전 서비스가 권한조회 호출) + REST |
| **search-service** | Kafka 구독 → ES 색인, 통합 검색 질의 | Elasticsearch | **GraphQL**(통합검색·필터 조합) + Kafka 소비 |
| **notification-service** | Kafka 구독 → 알림·활동 피드·(웹훅/메일) | PostgreSQL (or Redis) | REST/SSE + Kafka 소비 |

기존 gateway-server / eureka-server / auth-server / Keycloak은 그대로 재사용. 신규 5개.

> [!note] 왜 B안인가 — 세 가지 이유
> ① **CQRS의 구조적 실현**: wiki/alm은 쓰기 최적화(정규화 RDB), search-service는 읽기 최적화(ES 비정규화 색인). "DB 바쁘면 read/write 분리" 요구가 별도 장치 없이 구조로 해결된다 — 검색·목록성 조회 트래픽이 ES로 빠지면서 RDB 부하 자체가 줄어든다.
> ② **통신 3종의 적재적소**: 외부 API = REST, 내부 동기 호출(권한 체크·엔티티 참조) = gRPC, 상태 전파(색인·알림·활동·감사) = Kafka 이벤트. GraphQL은 "여러 도메인을 조합해 읽는 곳"(통합 검색·대시보드)에만 — 전면 GraphQL은 온프렘 운영 복잡도만 올린다.
> ③ **온프렘 설치 옵션화**: ES가 부담스러운 소규모 고객은 search-service를 Postgres FTS 폴백 모드로, notification은 꺼도 wiki/alm 코어는 동작하게 구성 가능.

### C안 — 세분할 (issue / project / comment / activity / attachment… 7개+)

팀별로 서비스를 소유하는 대기업 조직용 구조. 1인~소수 개발 + 온프렘 설치판에서는 배포·운영·버전 매트릭스만 폭발. **비추천.**

## 3. 통신 방식 매트릭스

| 경로 | 프로토콜 | 근거 |
|---|---|---|
| 프론트 → 게이트웨이 → 각 서비스 | REST | 범용성·디버깅 용이·캐시 친화 |
| 통합 검색·대시보드 조회 | GraphQL (search-service) | 다중 도메인 조합 조회, 필터·필드 선택 자유 |
| 서비스 간 동기 호출 (권한조회 등) | gRPC | 저지연·타입 안전(proto 계약)·스트리밍 |
| 상태 전파 (색인·알림·활동·감사) | **Redis Streams** 도메인 이벤트 (`EventPublisher` 추상화 뒤) | 비동기 결합 해제, 소비자 추가에 발행자 무수정 |

> [!important] 결정 변경 (2026-07-20, Wave B 착수 시): Kafka → Redis Streams
> 이 문서 초판과 B안 표는 상태 전파를 Kafka로 상정했으나, Wave B 착수 시 재검토에서 **Redis Streams로 변경**. 근거: ① 진짜 요구는 "색인·알림을 쓰기 경로에서 분리"이지 Kafka 자체가 아님, ② M 규모 이벤트량(초당 수십 건 미만)에 Kafka는 과설계 — 이 문서 스스로 S 티어 "Kafka 없어도 됨"을 인정, ③ Redis는 **이미 스택에 있어**(게이트웨이 rate limiter) 온프렘 설치판 컨테이너 +0, ④ 발행은 `EventPublisher` 인터페이스 뒤에 두므로 L 티어에서 Kafka 필요가 실재하면 구현체 교체로 국소 전환. 이하 본문·표의 "Kafka" 표기는 "이벤트 버스(Redis Streams)"로 읽을 것. Kafka는 L 티어 확장 경로로만 유지.

## 4. repo 구조 — 도메인 2 + 플랫폼 1

**repo는 코드 관리 경계, 서비스는 런타임 경계 — 둘은 일치할 필요가 없다.** platform-backend를 한 repo로 묶어도 안의 3서비스는 각각 별도 jar → 별도 이미지 → 별도 컨테이너로 배포된다.

묶기/분리 판별 질문: **"이것만 단독으로 설치·판매할 일이 있는가?"**

| 서비스 | 단독으로 의미가 있나 | 정체 → repo |
|---|---|---|
| wiki-backend | 예 — 위키만 쓰는 고객 가능 | **제품** → 독립 repo |
| alm-backend | 예 — 이슈트래커만 쓰는 고객 가능 | **제품** → 독립 repo |
| org / search / notification | 아니오 — 제품 없이는 빈 껍데기 | 플랫폼 부품 → platform-backend 멀티모듈 |

횡단 3개를 한 repo로 묶는 이유: ① 셋 다 **양쪽 제품을 섬기는 존재**라 어느 제품에도 소속 불가, ② **같은 계약(proto·이벤트 스키마)에 묶여 함께 변한다** — 이벤트 필드 하나 추가가 proto+search+notification 수정 한 세트인데, 같은 repo면 PR 하나로 원자적, 따로면 발행→버전범프→repo 3개 순회, ③ 신규 repo·CI가 6개 대신 3개.

```
MSA_TEMPLATE/                ← 우산 repo
├─ wiki-backend/             ← 독립 repo ★신규 — 제품 (단일 모듈로 시작)
├─ alm-backend/              ← 독립 repo ★신규 — 제품 (단일 모듈로 시작)
├─ platform-backend/         ← 독립 repo ★신규 — Gradle 멀티모듈
│   ├─ common-proto/         gRPC proto + Kafka 이벤트 스키마 → GitHub Packages 발행
│   ├─ org-service/
│   ├─ search-service/
│   └─ notification-service/
├─ gateway-server/ eureka-server/ auth-server/ board-service/  (기존 그대로)
└─ infra/                    compose가 전체 orchestrate
```

wiki·alm은 `common-proto`를 GitHub Packages 의존성으로 소비(`com.platform:common-proto:x.y.z`). GHCR 이미지 발행 CI 패턴에 jar 발행 스텝 하나 추가되는 것뿐. 각 제품 repo는 단일 모듈로 시작, 커지면 그 repo 안에서 멀티모듈로 분리.

런타임 예시 — 위키 페이지 저장: wiki-backend가 DB에 쓰고 `PageUpdated` 이벤트 발행 → search-service가 ES 색인, notification-service가 알림. **wiki는 search·notification의 존재를 모른다** → 소규모 설치에서 이 둘을 꺼도 코어는 동작.

## 5. 규모 티어 — 온프렘 설치 대상별 설계 차이

벤치마크: Confluence·Jira Data Center, GitLab 온프렘.

| | **S — 팀 (~50명)** | **M — 중소 조직 (~500명)** | **L — 대규모 (수천 명↑)** |
|---|---|---|---|
| 동시 사용자 | ~10명 | ~50명 (등록자의 ~10%) | 수백 명 |
| API 부하 | < 10 RPS | 수십 RPS, 피크 ~100 RPS | 수백~수천 RPS |
| 데이터량 | 문서·이슈 수만 건 | 문서·이슈 각 수십만 건, 첨부 수백 GB | 수백만 건, 첨부 수 TB |
| 검색 | Postgres FTS로 충분 | **ES 필수**, 검색 p95 < 200ms | ES 클러스터(3노드+), 샤딩 설계 |
| DB | 단일 인스턴스 | 단일 + 필요시 read replica 1 | R/W 분리 필수, 파티셔닝, 풀 튜닝 |
| Kafka | 없어도 됨 (단일 브로커) | 단일 브로커 | 3브로커 클러스터, 파티션 설계 |
| 배포 | compose, 서버 1대 (8c/16GB) | compose 또는 단일노드 k8s, 1~2대 (16c/32~64GB) | k8s 멀티노드, HA(Patroni·ES·Kafka 클러스터) |
| 고가용성 | 불필요 (백업만) | 선택 | 필수 |

> [!important] 온프렘 협업 도구의 진짜 병목
> 트래픽(RPS)보다 **데이터 누적량과 검색·집계 품질이 먼저 병목**이 된다. 사용자 500명이어도 RPS는 낮지만, 3년 치 문서 30만 건에서 "전문 검색 + 권한 필터링 + 하이라이팅"을 200ms에 돌려주는 게 진짜 과제. **ES 도입 판단 기준은 사용자 수가 아니라 문서량과 검색 요구 수준**이다.

### 제안 (미확정)

**M을 기본 설계 목표**로: 설계·코드는 M 기준(ES·Kafka 포함, 무상태 서비스로 수평 확장 가능)으로 하고, compose profile로 S(ES/Kafka 축소·FTS 폴백)·L(replica 추가) 양방향 조정. L 전용 요소(파티셔닝·HA 클러스터)는 확장 경로만 문서에 명시하고 초기 구현에서 제외.
