## 1️⃣ DB 선택 & 데이터 분할 (Choosing Database)

### ⚠️ The Shared Database Anti-pattern (공유 DB 안티패턴)
- 여러 서비스가 하나의 DB를 공유하는 것은 **피해야 할 나쁜 패턴**
- 공유 시 한 서비스의 스키마 변경이 다른 서비스를 망가뜨림 → 독립 배포·확장 불가
- 그래서 **Database-per-Service** 권장 (서비스마다 자기 DB 보유)

### Relational and NoSQL Databases
- 서비스 성격에 맞게 **RDBMS와 NoSQL을 골라 사용**
- 예: Product → Oracle, Cart/Payment → MySQL, User → Redis

### CAP Theorem (CAP 정리)
분산 시스템에서 다음 셋을 **동시에 다 만족할 수 없음 → 둘만 선택**
- **Consistency** (일관성)
- **Availability** (가용성)
- **Partition Tolerance** (분할 내성)

> 네트워크 장애(분할)는 피할 수 없으므로, 현실적으로 **일관성 vs 가용성** 사이의 트레이드오프가 된다.

### Data Partitioning (데이터 분할)

| 방식 | 기준 | 설명 |
|---|---|---|
| **Horizontal** (수평) | 행(row) | 사용자 1~100만 / 100만~200만 등으로 분할 (샤딩) |
| **Vertical** (수직) | 열(column) | 자주 쓰는 컬럼 / 안 쓰는 컬럼 분리 |
| **Functional** (기능) | 도메인 | 주문 데이터 / 결제 데이터 분리 |

### Database Sharding Pattern (샤딩)
- 큰 데이터를 여러 DB(샤드)에 쪼개 분산 저장
- 수평 분할을 물리적으로 여러 DB에 흩뿌림 → 단일 DB 용량·성능 한계 극복

## 2️⃣ 데이터 명령 / 조회 (Data Commands & Queries)

> 핵심 주제: **쓰기와 읽기를 어떻게 분리할 것인가**

### ⭐ CQRS Design Pattern
- Command Query Responsibility Segregation — **쓰기(Command)와 읽기(Query) 분리**
- **Write DB: RDBMS** — 데이터를 쓰는 곳, 정합성 중요
- **Read DB: NoSQL** — 데이터를 읽는 곳, 빠른 조회에 최적화
- 쓰기 DB 변경을 메시지 브로커로 `publish` → 읽기 DB가 `subscribe`해서 동기화

### Materialized View Pattern (구체화된 뷰)
- 조회를 빠르게 하려고 **미리 계산·가공한 데이터를 따로 저장**
- 읽을 때마다 복잡한 조인을 다시 하지 않도록 읽기 전용 뷰 생성

### Event Sourcing Pattern (이벤트 소싱)
- 현재 상태만 저장하는 대신, **상태를 바꾼 모든 이벤트(변경 이력)를 순서대로 저장**
- 이벤트를 재생하면 어느 시점의 상태든 복원 가능 (CQRS와 자주 함께 사용)

### Eventual Consistency Principle (최종 일관성)
- 쓰기 DB와 읽기 DB가 **그 순간엔 다를 수 있지만, 시간이 지나면 결국 일치**
- 즉시 일관성을 포기하는 대신 가용성·성능 확보 (CAP 트레이드오프와 연결)

## 🗺️ 다이어그램 한눈에

```text
클라이언트 → API Gateway → 각 서비스 (자기 DB 보유)
                                  │ 쓰기
                                  ▼
                          Write DB (RDBMS)
                                  │ publish (변경 이벤트)
                                  ▼
                          Message Broker
                                  │ subscribe
                                  ▼
                          Read DB (NoSQL) → 빠른 조회
```

> **한 줄 정리** — 쓰기는 정합성 좋은 **RDBMS**, 읽기는 빠른 **NoSQL**로 분리(CQRS)하고,
> 둘 사이를 **메시지 브로커**로 동기화하면서 **최종 일관성**을 유지한다.
