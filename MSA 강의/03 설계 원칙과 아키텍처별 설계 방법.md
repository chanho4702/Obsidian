# 🧭 설계 원칙 (Design Principles)

## 📋 전체 목록

| 원칙 | 의미 | 대응 요구사항 |
|---|---|---|
| **KISS** | Keep It Simple — 단순하게 설계 | Functional |
| **YAGNI** | You Aren't Gonna Need It — 불필요한 기능 배제 | Functional |
| **Separation of Concerns (SoC)** | 관심사 분리 | Functional |
| **SOLID** | 객체지향 설계 5원칙 | Functional |
| **Dependency Rule** | 의존성 방향 규칙 | Non-Functional |
| **Monolithic-First Strategy** | 모놀리식부터 시작 (필요 시 분리) | Non-Functional |
| **Separated UI** | UI 계층 분리 | Non-Functional |
| **Horizontal Scaling** | 수평 확장 | Non-Functional |
| **Load Balancer** | 부하 분산 | Non-Functional |

## 1️⃣ 기능 요구사항을 위한 원칙 (Functional Requirements)
기능 구현의 **명확성과 단순함**을 확보하기 위한 원칙들.

- **KISS** — 단순하게 설계
- **YAGNI** — 당장 필요 없는 기능은 만들지 않기
- **Separation of Concerns (SoC)** — 관심사 분리
- **SOLID** — 객체지향 설계 5원칙 (SRP·OCP·LSP·ISP·DIP)

🎯 **대상 기능 요구사항**
- 상품 목록보기
- 상품 검색 (제조사, 카테고리별)
- 장바구니 / 주문내역 확인
- 할인 쿠폰 등록
- 주문 / 결제

## 2️⃣ 비기능 요구사항을 위한 원칙 (Non-Functional Requirements)
**확장성·유지보수성·안정성** 등 시스템 품질을 확보하기 위한 원칙들.

- **Dependency Rule** — 의존성 방향 규칙 (안쪽 도메인은 바깥 인프라를 모르게)
- **Monolithic-First Strategy** — 모놀리식 우선 전략을 먼저 도입해서 검증
- **Separated UI** — UI 분리
- **Horizontal Scaling** — 수평 확장
- **Load Balancer** — 부하 분산

🎯 **대상 비기능 요구사항**
- 가용성 (Availability)
- 확장성 (Scalability)
- 동시 접속 사용자 처리 (대규모)
- 유지보수성 (Maintainability)
- 유연성 (Flexibility)
- 테스트 가능성 (Testability)
- 신뢰성 (Reliability)
- 재사용성 (Re-usability)

# 🏗️ 아키텍처별 설계 방법

## 1. Monolithic Architecture
모든 기능이 **하나의 코드베이스, 하나의 배포 단위**로 묶여 있는 구조.
UI · 비즈니스 로직 · 데이터 접근이 한 덩어리로 빌드·배포됩니다.

**👍 장점**
- 개발 초기 속도가 가장 빠름
- 함수 호출로 통신 → 디버깅·테스트가 쉬움
- 단일 DB → 데이터 일관성 유지가 쉬움

**👎 단점**
- 규모가 커지면 거대한 진흙덩어리(Big Ball of Mud)가 됨
- 한 군데 수정에도 전체 재배포 필요
- 일부 기능만 확장하는 것이 불가능

## 2. Modular Monolithic Architecture
여전히 **하나의 배포 단위**지만, 내부를 도메인별 모듈(`user/`, `order/`, `product/`, `payment/`)로
**명확히 경계 지어** 나눈 구조. 모놀리식과 마이크로서비스의 절충안.

**🎯 핵심 전략 — Monolithic-First Strategy**
> 처음부터 마이크로서비스로 가지 말고, 잘 모듈화된 모놀리식으로 시작해서 정말 필요해지면 그때 서비스를 떼어내라.

- 배포의 단순함(모놀리식)은 유지
- 모듈 분리의 장점(마이크로서비스)은 일부 확보
- 모듈 경계가 분명 → 나중에 분리하기 좋음
- **신뢰성(Reliability)**, **재사용성(Re-usability)** 강조

## 3. Microservice Architecture
기능을 **독립적으로 배포·실행되는 여러 작은 서비스**로 완전히 쪼갠 구조.
각 서비스가 자기 DB를 가지고(Database-per-Service) 네트워크(API/메시지)로 통신.

**🧩 핵심 패턴**
- Database-per-Service Pattern (서비스별 독립 DB)
- Decomposition Pattern (분해 패턴)

**⚡ 강조되는 비기능 요구사항**
- 높은 확장성 (High Scalability)
- 높은 가용성 (High Availability)
- 독립적 배포 가능
- 기술 스택 자유 선택
- 데이터 격리 (Data Isolation)
- 회복성 (Resilience)

**⚠️ 비용**
- 분산 시스템 특유의 복잡성 (네트워크 지연, 분산 트랜잭션)
- 운영 부담 증가

# 📊 한눈에 비교

| 구분 | 모놀리식 | 모듈러 모놀리식 | 마이크로서비스 |
|---|---|---|---|
| 배포 단위 | 1개 | 1개 | 서비스마다 개별 |
| 내부 구조 | 한 덩어리 | 도메인별 모듈 분리 | 독립 서비스로 분리 |
| 데이터베이스 | 단일 DB | 단일 DB | 서비스별 DB |
| 통신 방식 | 함수 호출 | 함수 호출 (모듈 간) | 네트워크 (API/메시지) |
| 확장성 | 전체만 가능 | 전체만 가능 | 서비스 단위 선택 확장 |
| 초기 개발 속도 | 빠름 | 빠름 | 느림 |
| 운영 복잡도 | 낮음 | 낮음 | 높음 |
| 적합한 시점 | 소규모 / 초기 | 중간 규모, 성장 대비 | 대규모 / 높은 트래픽 |

## ✅ 전체 흐름 정리
- **모놀리식** — 단순하지만 확장에 한계
- **마이크로서비스** — 확장·격리에 강하지만 복잡
- **모듈러 모놀리식** — 그 사이의 실용적 선택
  > "일단 모놀리식으로 단순하게 시작하되, 나중에 쪼갤 수 있도록 모듈 경계는 미리 잘 그어두자"

👉 그래서 많은 프로젝트가 **모듈러 모놀리식으로 시작**하는 것을 권장받습니다.
