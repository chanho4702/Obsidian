# 🟦 동기 통신 (Synchronous Communication)

## 💡 동기 통신이란?
요청을 보낸 쪽이 **응답이 올 때까지 기다리는** 방식.
- 클라이언트가 요청 → 응답을 받을 때까지 멈춰서 대기
- 주로 **클라이언트 ↔ 게이트웨이**, **서비스 ↔ 서비스** 사이에서 발생

## 1️⃣ 통신 프로토콜 (어떤 방식으로 주고받나)

| 프로토콜 | 특징 | 주 사용 구간 |
|---|---|---|
| **HTTP 기반 RESTful API** | 가장 보편적 | 클라이언트 ↔ 게이트웨이 |
| **GraphQL API** | 필요한 데이터만 골라 한 번에 조회 | 클라이언트 ↔ 서버 |
| **gRPC API** | 빠르고 효율적 | 서비스 ↔ 서비스 (내부) |
| **WebSocket API** | 실시간 양방향 통신 | 실시간 기능 |

- REST API를 통해 서비스를 호출한다.
- 서비스가 다른 서비스를 호출할 때도 REST API를 쓸 수 있으나, **내부 서비스 간 통신은 gRPC로 하는 것이 좋다** (빠르고 효율적).

## 2️⃣ 게이트웨이 라우팅 패턴 (요청을 어떻게 처리하나)
> 모든 클라이언트는 게이트웨이를 통해 서비스를 호출한다.

- **Gateway Routing Pattern** — 들어온 요청을 알맞은 서비스로 단순 전달(라우팅)
- **Gateway Aggregation Pattern** — 여러 서비스 호출 결과를 하나로 합쳐 한 번에 응답
- **Gateway Offloading Pattern** — 인증·로깅·SSL 등 공통 기능을 게이트웨이가 대신 처리해 서비스 부담 경감

## 3️⃣ 게이트웨이 구성 패턴 (게이트웨이를 어떻게 둘 것인가)
- **API Gateway Pattern** — 모든 요청의 단일 진입점(Single Entry Point). 클라이언트는 게이트웨이 하나만 알면 됨
- **Backends for Frontends (BFF)** — 클라이언트 종류별로 게이트웨이를 따로 구성 (웹용 / 모바일용 등)

## 🗺️ 전체 통신 흐름

```text
클라이언트 (웹 / 모바일)
      │  REST API
      ▼
  API Gateway  ← 단일 진입점
      │  (요청을 각 서비스로 분배)
      ▼
┌──────────────────────────────┐
│  Product  →  Oracle          │
│  Cart     →  MySQL           │
│  User     →  Redis           │
│  Order    →  DB              │
│  Payment  →  MySQL           │
└──────────────────────────────┘
      ▲
  서비스 간 내부 통신은 gRPC
```

### 📌 읽는 포인트
- **바깥쪽 (클라이언트 ↔ 게이트웨이)** → 범용적인 **REST API**
- **안쪽 (서비스 ↔ 서비스)** → 빠른 **gRPC**
- **각 서비스가 자기 전용 DB 보유** → Database-per-Service 패턴 ([[06 마이크로서비스 데이터 관리|상세]])
  - 서비스마다 용도에 맞는 DB를 자유롭게 선택 (Oracle, MySQL, Redis 등)

> **한 줄 정리** — 바깥은 호환성 좋은 **REST**, 안쪽은 빠른 **gRPC**로 역할을 나누고,
> **API Gateway**를 단일 진입점으로 두며, 서비스마다 **독립 DB**를 가진다.

# 🟪 비동기 통신 (Asynchronous Communication)

## 💡 비동기 통신이란?
요청을 보낸 쪽이 **응답을 기다리지 않고 바로 다음 일을 진행**하는 방식.

- 서비스끼리 직접 호출하지 않고 중간에 **메시지 브로커(Message Broker)** 를 둔다.
- 한쪽은 메시지를 던지고(**publish**), 다른 쪽은 가져가서 처리(**subscribe**)
- 보낸 쪽은 받는 쪽의 처리 시점을 신경 쓰지 않음
  → 서비스 간 결합이 느슨해지고, 한쪽이 느리거나 잠깐 죽어도 전체가 멈추지 않음

> 동기 통신과의 핵심 차이: **Message Broker**. 모든 서비스가 직접 연결되지 않고 브로커를 거친다.

**왜 쓰는가 — 최종 일관성 확보:**
분산 환경에서 서비스가 나뉘어 동작하다 보면 그 과정의 딜레이 때문에 데이터 정합성에 문제가 생길 수 있다.
이를 비동기 메시징으로 **최종적인 상태를 일치**시키기 위해 사용한다. (→ [[06 마이크로서비스 데이터 관리|최종 일관성]])

## 1️⃣ 통신 모델 (메시지를 누구에게 전달하나)

- **Single-receiver Communication (one-to-one)**
  - 메시지 하나를 **한 곳**만 받음
  - 보통 큐(Queue) 방식 — "누군가 하나가 처리하면 되는" 작업
  - 메시지 브로커만 서비스 통신을 처리하며, 서비스가 다른 서비스에 직접 통신 요청하지 않는다

- **Multiple-receiver Communication (one-to-many, topic)**
  - 메시지 하나를 **여러 곳**이 동시에 받음
  - 토픽(Topic) 방식 — 예: "주문 완료" 이벤트를 결제·재고·알림 서비스가 각각 받아 처리

- **Dependency Inversion Principle (DIP)** — 의존 역전 원칙
  - 서비스가 서로의 구체적 구현을 직접 알지 않고 **브로커(추상화된 메시지 계약)에만 의존**
  - 모듈 간 결합도를 낮추고, 모든 것을 추상화하여 사용
  - 코드의 유연성·테스트 용이성이 좋아지고, 서비스 간 통신의 의존성이 줄어든다

## 2️⃣ 메시징 패턴 (메시지를 어떻게 흘려보내나)
- **Fan-Out Publish/Subscribe Pattern** — 하나의 메시지를 구독 중인 여러 서비스로 한 번에 퍼뜨림 (one-to-many 대표 패턴)
- **Topic-Queue Chaining Pattern** — 토픽과 큐를 연결 → 브로드캐스트(토픽) + 작업 분배(큐) 장점을 함께 사용
- **Load Balancing Pattern** — 같은 메시지를 처리할 인스턴스가 여럿일 때 부하를 고루 분배

## 🗺️ 전체 흐름

```text
서비스 (Product, Cart, User, Order, Payment …)
      │  publish (메시지를 던짐)
      ▼
  Message Broker  ← 중간 매개체
      │  subscribe (메시지를 가져감)
      ▼
다른 서비스가 받아서 처리
```

## 🔁 동기 vs 비동기 비교

| 구분 | 동기 (Sync) | 비동기 (Async) |
|---|---|---|
| 응답 대기 | 기다림 | 안 기다림 |
| 중간 매개체 | API Gateway (직접 호출) | Message Broker (publish/subscribe) |
| 서비스 결합도 | 상대적으로 높음 | 느슨함 (loose coupling) |
| 한쪽 장애 시 | 영향 받기 쉬움 | 영향 격리됨 |
| 대표 기술 | REST, gRPC | Kafka, RabbitMQ 등 |

> **한 줄 정리** — 서비스끼리 직접 부르지 않고 **메시지 브로커**를 통해 publish/subscribe 한다.
> 덕분에 결합이 느슨해지고 장애가 격리되어, 확장성과 회복성이 높아진다.

# 🧱 보충: Dependency Inversion 원칙 (DIP)

- 코드의 재사용성 향상, 테스트가 간단해짐, 확장성에도 좋음
- 모듈 간의 결합도가 낮아짐
- **결합도를 낮추고 응집도를 높이는 작업**이 중요하다

### ❌ 나쁜 예 — 구체 클래스에 직접 의존
`NotificationService`(고수준)가 `EmailService`(저수준)에 직접 의존 → 이메일에 종속적

```java
class EmailService {
    public void sendMail(String message) {
    }
}

class NotificationService {
    private EmailService emailService = new EmailService();

    public void notify(String message) {
        emailService.sendMail(message); // 이메일에 강하게 의존
    }
}
```

### ✅ 좋은 예 — 추상 인터페이스에 의존
`NotificationService`와 `EmailService` 모두 `MessageService` 인터페이스에 의존
→ 이메일이든 SMS든 구현을 갈아 끼워도 무관

```java
interface MessageService {
    void sendMessage(String message);
}

class EmailService implements MessageService {
    public void sendMessage(String message) {
        // 이메일 보내는 로직
    }
}

class SMSService implements MessageService {
    public void sendMessage(String message) {
        // SMS 보내는 로직
    }
}

class NotificationService {
    private MessageService messageService;

    public NotificationService(MessageService messageService) {
        this.messageService = messageService; // 주입받음
    }

    public void notify(String message) {
        messageService.sendMessage(message); // 어떤 구현이든 무상관
    }
}
```
