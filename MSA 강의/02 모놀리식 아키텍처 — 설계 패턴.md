## 📐 주요 설계 패턴 (Design Patterns)

### 1. Layered (N-Layer) Architecture
계층별로 책임을 분리한 구조. 각 계층은 **바로 아래 계층에만 의존**한다.

- **Presentation Layer (UI)** — 사용자 인터페이스
- **Business Layer (Domain Logic)** — 비즈니스 로직 처리
- **Persistence Layer (CRUD)** — 데이터 영속성 / CRUD
- **Infrastructure Layer** — 보안(Security), 외부 연동(External)

### 2. MVC (Model-View-Controller)
요청 처리 흐름 (Model → data). 사용자의 요청이 계층을 따라 흐른다.

```text
Client → Controller → Service → Repository → Database
```

- **Controller** — 요청을 받아 적절한 Service로 위임
- **Service** — 비즈니스 로직 수행
- **Repository** — DB 접근(ORM으로 매핑)

### 3. Modular Monolithic
기능별 디렉토리 구조 정의. **하나의 배포 단위지만 내부는 도메인별로 분리**한다.

- 도메인 단위로 디렉토리 분리
- 예) `user/` `order/` `product/` `payment/`
- 모듈 경계를 미리 그어두면 → 나중에 마이크로서비스로 분리하기 쉬움

### 4. Database 설계
- 정규화된 테이블 설계 (RDBMS)
- **주요 엔티티:** `User`, `Product`, `Order`
- 주문 상태를 트랜잭션으로 관리

| 주문 상태 | 설명 |
|---|---|
| `Placed` | 주문 완료 |
| `Shipped` | 배송 중 |
| `Delivered` | 배송 완료 |
| `Canceled` | 주문 취소 |

### 5. API 설계 및 문서화
- **RESTful API 설계** — `POST` / `GET` / `PUT` / `DELETE`
- **API 문서화** — Swagger 또는 Postman 활용

### 6. 보안 및 인증 적용
- **인증 / 인가** — JWT(JSON Web Token) 또는 OAuth2 활용
- **통신 보안** — HTTPS 적용
- **CORS 정책 설정**

### 7. 성능 최적화 전략
- **Redis Caching 적용** — 장바구니, 주문 내역 캐싱
  - DB에 저장하기 전에 캐시에 먼저 담아두어 응답 속도를 높이고 DB 부하를 줄인다
- **데이터베이스 인덱싱 및 쿼리 최적화**
- **Lazy Loading & Pagination 적용**
  - 필터링·페이징이 필요할 때마다 그때그때 필요한 만큼만 데이터를 로드 → 초기 로딩 부담 감소

## ⚖️ Modular Monolithic 장단점

> Modular Monolithic 아키텍처 선택에 따른 장점과 단점 정리입니다.

| ✅ 장점 (Pros) | ⚠️ 단점 (Cons) |
|---|---|
| 개발 속도가 빠름 | 규모가 커지면 유지보수가 어려워짐 |
| 배포가 단순함 | 배포 리스크가 큼 |
| 디버깅 및 테스트가 쉬움 | 확장성 제한 |
| 데이터 일관성 유지가 쉬움 | 팀 협업 시 비효율 발생 |

### 모놀리식의 근본 특성
- 하나의 데이터베이스에 모든 작업을 진행함 → 데이터 일관성은 쉽지만, DB가 병목이 됨
- 전체 애플리케이션을 한 번에 배포해야 함 → 작은 수정에도 전체 재배포 필요

> 📷 *(강의 다이어그램: 단일 코드베이스 + 단일 DB 구조 — Notion 원본 참조)*
