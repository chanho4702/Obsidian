# 🔀 마이크로서비스 트랜잭션 관리 (Transaction Management)

# 📦 분산 트랜잭션 (Distributed Transactions)

## 💡 왜 어려운가?
- 모놀리식 → DB가 하나라 트랜잭션 하나로 묶으면 끝 (전부 성공 or 전부 롤백)
- 마이크로서비스 → **서비스마다 DB가 따로** → 전통적 DB 트랜잭션으로 묶을 수 없음
- 예: "주문 생성 → 재고 차감 → 결제"가 서로 다른 서비스/DB에서 발생.
  결제만 실패하면 앞의 주문·재고는 어떻게 되돌릴까?

## ⭐ 1️⃣ SAGA Pattern
분산 트랜잭션을 **여러 개의 작은 로컬 트랜잭션 + 보상 단계**로 쪼개 처리.
중간에 실패하면 이미 끝난 작업들을 **보상 트랜잭션으로 되돌림**.

| 방식 | 설명 | 특징 |
|---|---|---|
| **Choreography** | 중앙 조율자 없이 각 서비스가 이벤트를 듣고 알아서 반응 | 느슨함 / 흐름 추적 어려움 |
| **Orchestration** | 중앙 조율자가 단계별로 지휘 | 흐름 명확 / 조율자에 집중 |

## 2️⃣ Compensating Transaction Pattern (보상 트랜잭션)
- 이미 커밋된 작업을 **취소하는 반대 작업** (예: 결제 취소, 재고 복구)
- 롤백이 불가능한 분산 환경에서 되돌리는 동작을 명시적으로 실행
- SAGA에서 실패 시 작동

## 3️⃣ Dual-Write Problem (이중 쓰기 문제)
- **DB 저장**과 **메시지 브로커 이벤트 발행**을 동시에 해야 할 때 발생하는 문제
- 하나는 성공, 하나는 실패하면 데이터와 이벤트가 어긋남
- 근본 원인: 두 작업은 서로 다른 시스템 → 하나의 트랜잭션으로 묶을 수 없음

## 4️⃣ Transactional Outbox Pattern (아웃박스)
- 이중 쓰기 문제의 해법
- 이벤트를 브로커에 직접 쏘지 않고, **같은 DB 안의 "Outbox" 테이블에 비즈니스 데이터와 함께 한 트랜잭션으로 저장**
- DB 저장 성공 시 이벤트도 무조건 Outbox에 남음 → 어긋날 수 없음
- (다이어그램: 하나의 DB에 `Order` 테이블 + `Outbox` 테이블 공존)

## 5️⃣ CDC (Change Data Capture)
- DB의 **변경 사항을 감지해서 이벤트로 흘려보내는 기법**
- Outbox 테이블에 쌓인 변경분을 읽어 브로커로 전달
- 대표 도구: **Debezium** (Log-based CDC — DB 로그를 읽는 방식)

### 🗺️ Outbox + CDC + SAGA 흐름

```text
Order 서비스가 주문 처리
      │  하나의 트랜잭션으로 같이 저장
      ▼
┌─────────────────────────────┐
│  Order 테이블 │ Outbox 테이블 │  ← 한 DB 안
└─────────────────────────────┘
      │  Debezium이 변경 감지 (Log-based CDC)
      ▼
  Message Broker  ──publish──▶ Stock/Payment 등이 subscribe
      (SAGA Pattern으로 다음 단계 연쇄 처리)
```

> **세 가지가 맞물린다:** Outbox로 데이터·이벤트를 안전하게 저장 → CDC(Debezium)가 변경 감지해 브로커로 전달 → SAGA로 여러 서비스 거래를 단계적으로 진행/보상.

# ⚡ 이벤트 기반 아키텍처 (EDA)

## 💡 EDA란?
- **Event-Driven Architecture** — 서비스가 직접 호출하지 않고, **이벤트 발생을 중심으로 시스템이 반응**하도록 설계
- 앞의 SAGA/Outbox가 동작하는 **바탕이 되는 이벤트 중심 인프라**

## 핵심 개념
- **Asynchronous, Decoupled communication** — 비동기·느슨한 결합. 서비스끼리 직접 모르고 이벤트로만 소통
- **Event Hubs** — 모든 이벤트가 모이고 흐르는 중앙 통로
- **Stream-Processing** — 이벤트를 흐르는 스트림으로 연속 처리
- **Real-time processing** — 실시간 처리
- **High volume events** — 대량 이벤트 처리에 적합

## 🗺️ 다이어그램 핵심

```text
Product  Cart  User  Order  Payment   ← 모든 서비스
   │     │     │     │       │
   ▼     ▼     ▼     ▼       ▼
┌──────────────────────────────────┐
│   Event Hub / Event Backbone     │  ← 이벤트가 순서대로 로그처럼 쌓임
└──────────────────────────────────┘
   (필요한 서비스가 여기서 이벤트를 가져감)
```

- 가운데 큰 막대 = **Event Hub / Event Backbone (이벤트 중추)**
- 모든 서비스가 이벤트를 이 중추로 흘려보내고, 필요한 서비스가 가져감
- 칸칸이 쌓이는 모양 = 이벤트가 **순서대로 로그처럼 적재** (Kafka 같은 스트림 플랫폼 구조)

> **Message Broker**가 메시지를 전달하는 우체통이라면,
> **Event Backbone**은 모든 이벤트가 흐르는 **중앙 신경망**에 가깝다.

---

## ✅ 전체 한 줄 정리
> 분산 트랜잭션은 **SAGA + Outbox + CDC** 조합으로 안전하게 처리하고,
> 이 모든 것은 **EDA(이벤트 중추)** 위에서 비동기·느슨한 결합으로 돌아간다.
