# 🛡️ 마이크로서비스 회복성·관측성·모니터링

> "한 서비스가 장애를 일으켜도 전체가 무너지지 않게, 그리고 문제를 빨리 찾아내게 하려면?"

## 1️⃣ 회복성 패턴 (Resilience Patterns)
서비스 호출이 실패하거나 느려질 때 **장애가 옆 서비스로 번지지 않도록** 막는 패턴들.
(예시 설정값은 실제 적용 시 튜닝 대상)

| 패턴 | 설명 | 예시 설정 |
|---|---|---|
| **Retry** (재시도) | 일시적 실패면 몇 번 다시 시도 | `Retries=3` |
| **Circuit-Breaker** (서킷 브레이커) | 계속 실패하면 회로를 끊어 호출 자체를 차단 → 무한 재시도로 인한 추가 붕괴 방지 | `ExceptionAllowed=20`, `CheckCircuit=30 sec` |
| **Bulkhead** (격벽) | 자원(스레드/커넥션)을 서비스별로 격리 → 한 곳이 자원을 다 써도 다른 곳은 멀쩡 | — |
| **Timeout** (타임아웃) | 응답이 너무 오래 걸리면 기다리지 않고 끊음 | `Duration=5 sec` |
| **Fallback** (폴백) | 실패 시 대체 응답 제공 (예: 추천 실패 시 기본 목록) | — |

## 2️⃣ 관측성 & 모니터링 (Observability & Monitoring)
분산된 수많은 서비스에서 "지금 어디서 무슨 일이 일어나는지" 파악하는 도구들.

### Distributed Logging (분산 로깅)
- 여러 서비스에 흩어진 로그를 **한 곳에 모음 (Log Aggregation)**
- **fluentd**가 각 서비스 로그를 수집·집계

### Distributed Tracing (분산 추적)
- 하나의 요청이 여러 서비스를 거칠 때 **전체 경로를 추적**
- **TraceID** — 요청 전체에 부여되는 ID
- **SpanID** — 각 서비스 구간마다 부여되는 ID
- → "느린 요청이 정확히 어느 서비스 구간에서 지연됐는지" 파악 가능

### 대표 도구 조합

| 조합 | 용도 |
|---|---|
| **Elastic Stack** = ElasticSearch + Logstash + Kibana (ELK) | 로그 수집·저장·검색·시각화 (분산 로깅) |
| **OpenTelemetry + Zipkin** | 추적 데이터 수집 표준 + 추적 시각화 (분산 추적) |
| **Prometheus + Grafana** | 지표(metric) 수집 + 대시보드 시각화 (모니터링) |

## 🗺️ 다이어그램 한눈에

```text
요청 (TraceID 부여)
   │
API Gateway → Product (SpanID) → Cart (SpanID) → ...
   │   각 서비스에 Retry / Circuit-Breaker / Timeout 등 적용
   ▼
각 서비스 로그 → fluentd (Log Aggregation) → 한 곳에 모임
```

---

## ✅ 한 줄 정리
> 왼쪽(Retry·Circuit-Breaker·Timeout)은 **장애를 견디는 패턴**,
> 오른쪽(TraceID·SpanID·fluentd)은 **장애를 관측·추적하는 체계**.
> → "장애가 나도 버티고(Resilience), 어디서 났는지 빨리 찾는다(Observability)"
