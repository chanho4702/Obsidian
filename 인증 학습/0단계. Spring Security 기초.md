# 0단계. Spring Security 기초

## 1. HTTP Stateless — 모든 문제의 출발점

HTTP는 기본적으로 **상태를 기억하지 않는(stateless)** 프로토콜이다. 요청 하나가 끝나면 서버는 그 클라이언트가 누구였는지 완전히 잊어버린다. 즉 요청 A에서 로그인을 했어도, 다음 요청 B에서는 서버 입장에서 "넌 누구야?" 상태로 돌아간다.

로그인 같은 기능은 "이 사람이 아까 로그인한 그 사람"이라는 걸 계속 알아야 한다. Stateless인데 어떻게 상태를 유지할까? → 이걸 해결하려고 **Cookie**와 **Session**이 나왔다.

## 2. Cookie — 클라이언트가 들고 다니는 신분증

쿠키는 서버가 클라이언트(브라우저)에게 발급해서 저장시키는 작은 데이터다. 브라우저는 이후 같은 서버로 요청할 때마다 이 쿠키를 자동으로 함께 보낸다. 그래서 서버는 쿠키를 보고 "아, 아까 그 사람이구나"를 판단할 수 있다.

- 핵심은 **저장 위치가 클라이언트**라는 점.
- 쿠키에 민감한 정보(비밀번호 등)를 직접 담으면 위험하다.
- 그래서 보통 쿠키엔 식별자만 담고, 세션을 같이 쓴다.

## 3. Session — 서버가 진짜 정보를 보관하는 곳

세션은 **저장 위치가 서버**다. 동작 방식:

1. 로그인 성공 → 서버가 세션을 생성하고 그 안에 사용자 정보를 저장
2. 세션을 식별하는 ID(`JSESSIONID`)를 쿠키로 클라이언트에 전달
3. 이후 요청마다 클라이언트는 이 세션 ID 쿠키를 보냄
4. 서버는 ID로 자기가 보관 중인 세션을 찾아 사용자 정보를 복원

쿠키(식별자) + 세션(실제 정보) 조합으로 stateless HTTP 위에서 상태를 유지한다. Spring Security도 기본값은 이 세션 방식으로 인증 상태를 유지한다.

## 4. Authentication (인증) — "너 누구야?"

인증은 **신원을 확인하는 과정**이다. 아이디/비밀번호가 맞는지 검증해서 "이 사람은 진짜 홍길동이다"를 확정하는 단계. 로그인이 바로 인증이다.

## 5. Authorization (인가/권한) — "넌 뭘 할 수 있어?"

인가는 인증된 사용자가 **특정 자원에 접근할 권한이 있는지** 확인하는 과정이다. 예: 로그인은 했지만(인증 O), 관리자 페이지는 ADMIN 권한이 있어야만 들어갈 수 있다(인가 X).

> 순서가 중요하다: **인증이 먼저, 인가가 나중.** 누구인지 모르면 권한도 따질 수 없다.

## 6. SecurityContext — 인증 정보가 담기는 보관함

인증이 끝나면 그 결과(누가 로그인했고, 어떤 권한을 가졌는지)를 담아둬야 요청 처리 내내 꺼내 쓸 수 있다. 그 보관함이 `SecurityContext`다.

구조:

```
SecurityContextHolder → SecurityContext → Authentication
                                            ├─ Principal (사용자 정보)
                                            └─ Authorities (권한)
```

- `SecurityContextHolder`는 보통 `ThreadLocal` 기반 → **현재 요청을 처리하는 스레드 어디서든** 인증 정보를 꺼낼 수 있다.
- 접근 방법: `SecurityContextHolder.getContext().getAuthentication()`

## 7. SecurityFilterChain — Spring Security의 심장

Spring Security의 동작은 결국 **필터(Filter)들의 연쇄**로 이루어진다. 모든 HTTP 요청이 컨트롤러에 도착하기 전에, 여러 개의 보안 필터를 순서대로 통과한다. 이 필터들의 묶음이 `SecurityFilterChain`이다.

대표 필터 흐름:

- `UsernamePasswordAuthenticationFilter` (로그인 인증 처리)
- → `AuthenticationManager`에게 인증 위임
- → `AuthorizationFilter` (접근 권한 검사)
- → 통과하면 컨트롤러로

필터 체인은 실제 서블릿 필터로 등록되며, `DelegatingFilterProxy`와 `FilterChainProxy`를 통해 스프링 빈으로 관리되는 보안 필터들이 끼어드는 구조다.

## 전체 동작 원리 (요청 흐름)

```
        HTTP 요청
   (쿠키: JSESSIONID 포함)
            │
            ▼
┌─ SecurityFilterChain ──────────────┐
│                                    │
│   인증 필터 (Authentication)        │
│   신원 확인 · 로그인                │
│            │                       │
│            ▼                       │
│   SecurityContext                  │
│   인증 정보 저장                      │
│            │                       │
│            ▼                       │
│   인가 필터 (Authorization)          │
│   접근 권한 검사                      │
│                                    │
└────────────│───────────────────────┘
             ▼
        Controller
      (비즈니스 로직 실행)
```

## 핵심 한 줄 정리

> **HTTP는 상태가 없으니(Stateless), 쿠키와 세션으로 "누구인지"를 유지하고, 요청이 들어오면 SecurityFilterChain의 필터들이 차례로 인증(Authentication)으로 신원을 확인해 SecurityContext에 담은 뒤, 인가(Authorization)로 권한을 검사하고 통과하면 컨트롤러로 넘긴다.**

기억할 포인트 3가지:

1. **인증 → 인가 순서**는 절대 안 바뀐다. 누구인지 알아야 권한도 따질 수 있다.
2. 인증 결과는 **항상 SecurityContext에 저장**되고, 세션과 연동돼 다음 요청에서도 유지된다.
3. Spring Security의 모든 동작은 결국 **필터 체인**이다. "어떤 필터가 언제 끼어드는가"를 이해하면 나머지는 따라온다.
