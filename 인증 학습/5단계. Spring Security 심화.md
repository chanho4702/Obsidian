# 5단계. Spring Security 심화 — 직접 만든 걸 표준 부품으로

> [!abstract] 한 줄
> 5단계는 "전부 외워서 백지에서 짜는" 단계가 아니라, **[[4단계. 토큰 보안 (Access·Refresh)|4단계]]에서 직접 만든 인터셉터/검증을 Spring Security의 표준 부품으로 갈아끼우는 단계**다. 밑바닥 원리는 3·4단계에서 이미 익혔으니, 여기선 그 지식을 **프레임워크 관용구에 매핑**한다.

실습: `C:\myStudy\springSecurity` — `tokenSecurity`(4단계)를 이관. 토큰 메커니즘 자체(`JwtProvider`·`RefreshTokenStore`·`AccessTokenBlacklist`·회전·재사용탐지·블랙리스트)는 **그대로 유지**하고, 검증 경로만 Security로 옮긴다.

## 1. 4단계 인터셉터 → 5단계 부품 3개로 분해

4단계의 `HandlerInterceptor` 하나가 5단계에선 세 부품으로 흩어진다:

| 4단계 | 5단계 | 역할 |
|---|---|---|
| 인터셉터의 토큰 검증 | **`JwtAuthenticationFilter`** (`OncePerRequestFilter`) | 토큰 꺼내 검증 → `SecurityContext`에 신원 주입 |
| 인터셉터의 경로 분기 | **`SecurityConfig`** (`SecurityFilterChain`) | 어느 경로가 `permitAll`/`authenticated`인지 |
| 인터셉터의 401 응답 | **`AuthenticationEntryPoint`** | 인증 실패 시 JSON 401 응답 |

## 2. `JwtAuthenticationFilter` 한 줄씩

```java
@Component
public class JwtAuthenticationFilter extends OncePerRequestFilter { ... }
```

- **`@Component`** — 빈 등록. 그래야 `SecurityConfig`가 생성자 주입으로 받아 `addFilterBefore(...)`로 체인에 끼운다.
- **`extends OncePerRequestFilter`** — ⭐ 5단계 핵심. 4단계 `HandlerInterceptor`를 대체. 이름대로 **요청 1번당 딱 1번** 실행 보장(순수 `Filter`는 포워드/디스패치 때 여러 번 탈 수 있음).
- 의존성 3개: **JWT 검증기**(3·4단계 그대로), **블랙리스트**(Redis, 4단계 그대로), **EntryPoint**(인터페이스로 받고 구현은 스프링이 주입 = 느슨한 결합).

**처리 흐름:**
1. **헤더 없으면?** → ⭐ 4단계와 결정적으로 다른 철학. 4단계 인터셉터는 헤더 없으면 즉시 `MALFORMED`를 던졌다. 5단계 필터는 **아무것도 안 하고 다음 필터로 넘긴다**(`SecurityContext`가 빈 채로 진행).
   - 그럼 누가 막나? → 뒤의 **`AuthorizationFilter`**가 "이 경로는 `authenticated()`인데 인증 정보 없네?" 하고 EntryPoint(401)로 보냄.
   - 왜? `permitAll` 경로(`/`, `/api/auth/login`)는 토큰이 없는 게 정상. 필터가 무조건 막으면 로그인 페이지도 못 본다 → "막을지 말지"는 경로를 아는 `SecurityConfig`에 위임(역할 분리).
2. **토큰 있으면 검증:**
   - 2-a) `JwtProvider.verify()` 그대로 호출. 위조/만료면 `JwtException` → `catch`로.
   - 2-b) ⭐ 서명·만료가 멀쩡해도 **로그아웃된 토큰이면 거절**. `jti`가 Redis 블랙리스트에 있으면 `REVOKED`. ("순수 stateless → 필요할 때만 stateful" 트레이드오프 지점, 4단계 개념 그대로)
3. **신원 객체 만들기:**
   - ⭐ `"ROLE_" + role` — Spring 관례. 토큰엔 `"USER"`로 담겨도 권한은 `"ROLE_USER"`로 만들어야 `@PreAuthorize("hasRole('ADMIN')")`의 `hasRole`(자동으로 `ROLE_` 붙임)과 맞는다. `SimpleGrantedAuthority`로 표현.
   - ⭐⭐ `new UsernamePasswordAuthenticationToken(principal, null, authorities)` — **3-인자 생성자**를 쓰면 "이미 인증 완료" 상태가 된다. credentials는 `null`(토큰으로 이미 증명, 비번 메모리에 안 남김). (2-인자면 `authenticated=false` 미인증 — **3-인자 = 인증됨**이 핵심 차이.)
   - `SecurityContextHolder.getContext().setAuthentication(...)` = "이 요청은 인증된 사용자다"를 스프링 전체에 선언. 이후 `AuthorizationFilter`·`@PreAuthorize`·컨트롤러의 `Authentication` 파라미터가 전부 여기서 꺼내 쓴다.
4. **검증 실패 시(`catch`):** 오염됐을지 모를 컨텍스트를 방어적으로 비우고, 실패 코드(`EXPIRED`/`BAD_SIGNATURE`/`REVOKED`/`MALFORMED`)를 `request.setAttribute`로 "놓아둠" → EntryPoint가 같은 키로 읽어 JSON에 담는다. (필터와 EntryPoint는 직접 호출 안 하고 request 속성으로 정보를 주고받음 → 키를 상수로 공유)

> `filterChain.doFilter(...)`를 **직접 호출**해야 다음 필터로 넘어간다. 4단계 인터셉터는 `return true`로 통과를 표현했지만, 필터는 호출 안 하면 거기서 체인이 끊긴다.

## 3. 401 vs 403 (인증 vs 인가)

| 코드 | 의미 | 담당 |
|---|---|---|
| **401** | "누군지 모름" (인증 실패) | `AuthenticationEntryPoint` |
| **403** | "누군진 아는데 권한 부족" (인가 실패) | `AccessDeniedHandler` |

- 데모: `alice/password`(USER), `admin/admin`(ADMIN). USER가 관리자 API 호출 → **403**. 토큰 없이 보호 API → **401**.

## 4. Method Security

- `@PreAuthorize("hasRole('ADMIN')")`로 메서드 단위 권한. 화면에 **⑤ 권한(Role) 기반 접근** 카드로 추가.
- 내부적으로 `DaoAuthenticationProvider`가 `UserDetailsService` + `PasswordEncoder`로 비번 대조(이 Provider는 직접 안 만들고 스프링이 자동 구성).

## 🎯 어디까지 학습해야 하나

> **5단계는 "직접 만든 걸 표준 부품으로 갈아끼우는 손맛 + 401/403/Method Security 개념"까지면 충분.** 프레임워크 내부 소스까지 팔 필요는 없다.

**외울 필요 없는 것**: 필터 체인의 전체 기본 필터 순서(`SecurityContextHolderFilter`, `ExceptionTranslationFilter`, `AuthorizationFilter`...)는 부팅 로그에 찍히는 목록일 뿐 — 순서를 외우지 말고 **"내 필터가 그 사이 어디 들어가는지"**만 감 잡으면 OK. `AuthenticationConfiguration.getAuthenticationManager()` 빈 노출 보일러플레이트는 패턴으로 복붙해도 무방.

### 자가 점검 (말로 설명되면 통과)
- 4단계 인터셉터가 5단계에서 **어떤 부품 3개**로 흩어졌나? (필터 / SecurityConfig / EntryPoint)
- 검증 통과한 토큰의 신원이 컨트롤러까지 어떻게 전달되나? (`SecurityContext` → `Authentication` 주입)
- 401과 403은 각각 누가 처리하나?

## 검증 결과
- 부팅 로그: 필터 체인에 `JwtAuthenticationFilter` 정상 삽입, `AuthenticationManager`가 `customUserDetailsService`로 구성 확인.
- 틀린 비번 → **401 BAD_CREDENTIALS**, 올바른 비번 → 인증 통과(RT 발급 단계에서만 Redis 필요). → Spring Security 인증 체인 완전 동작 확인.
