# 1단계. 세션 기반 로그인 (Session Login)

> [!abstract] 한 줄 요약
> 로그인하면 서버가 세션을 만들고(Session), 그 열쇠인 세션 ID를 쿠키(Cookie)에 담아 주고받는다. Java에서는 그 쿠키 이름이 **JSESSIONID**이며, 이 쿠키 자동 전송을 악용하는 공격이 **CSRF**다.

---

## 📚 1단계 학습 — 핵심 개념

### 1. Session Login (세션 기반 로그인)

핵심 아이디어: **"서버가 로그인 상태를 기억한다"**

- HTTP는 원래 **상태가 없는(stateless)** 프로토콜 → 요청이 끝나면 서버는 누구였는지 잊어버림.
- 로그인 성공 시 서버가 메모리(또는 DB/Redis)에 사용자 정보를 담은 **세션(session)** 을 만들고, 고유한 **세션 ID** 를 붙임.
- 클라이언트(브라우저)에게는 **세션 ID만** 쿠키로 전달.
- 이후 요청마다 세션 ID를 함께 보내면, 서버가 "아까 로그인한 그 사람"을 알아봄.

> **핵심 구분**
실제 사용자 정보는 **서버**가 들고 있고, 클라이언트는 **열쇠(세션 ID)** 만 가진다.
→ JWT(토큰) 방식은 사용자 정보 자체를 클라이언트가 들고 다닌다는 점에서 다름.
> 

### 2. Cookie (쿠키)

쿠키 = 서버가 브라우저에 저장해두는 작은 데이터 조각 (세션 ID의 "운반 수단")

동작은 HTTP 헤더 2개로 이뤄짐:

| 시점 | 헤더 | 예시 |
| --- | --- | --- |
| 서버 응답 시 | `Set-Cookie` | `Set-Cookie: JSESSIONID=abc123...` |
| 브라우저 다음 요청 시 | `Cookie` (자동) | `Cookie: JSESSIONID=abc123...` |

> 이 **"자동으로 붙는다"** 는 점이 쿠키의 편의이자, CSRF 공격의 빌미가 된다.
> 

**주요 보안 속성**

| 속성 | 역할 |
| --- | --- |
| `HttpOnly` | JS에서 `document.cookie`로 못 읽게 막음 → XSS로 세션 ID 탈취 방지 |
| `Secure` | HTTPS 연결에서만 쿠키 전송 |
| `SameSite` | 다른 사이트발 요청에 쿠키를 붙일지 제어 (`Strict`/`Lax`/`None`) → CSRF 방어 |
| `Max-Age` / `Expires` | 쿠키 수명 (없으면 브라우저 닫을 때 사라지는 세션 쿠키) |

### 3. JSESSIONID

!session_login_flow.svg

- **Java 서블릿 컨테이너(Tomcat 등)가 세션을 식별하려고 자동 생성하는 쿠키 이름.**
- 위에서 말한 "세션 ID"의 Java 진영 기본 이름.
- `HttpSession`을 사용하거나 로그인 처리를 하면 톰캣이 알아서 `Set-Cookie`로 내려줌.
- 다른 언어의 동일 개념: PHP = `PHPSESSID`, Express = `connect.sid`.

> **세션 고정(Session Fixation) 공격 방어**
로그인 성공 시점에 세션 ID를 **새로 발급**해야 한다. Spring Security는 기본으로 처리해 줌.
> 

### 4. CSRF (Cross-Site Request Forgery, 사이트 간 요청 위조)

세션 방식의 **편의성(쿠키 자동 첨부) 때문에 생기는 취약점.**

**공격 시나리오**

1. 사용자가 은행 사이트에 로그인한 상태(세션 살아있음).
2. 공격자가 만든 악성 페이지 방문 → 페이지에 숨겨진 요청: `<img src="https://bank.com/transfer?to=공격자&amount=100만">`
3. 브라우저는 bank.com 요청이니 **세션 쿠키를 자동으로** 붙여 전송.
4. 서버는 정상 로그인 사용자의 요청으로 인식 → 이체 실행.

→ **사용자가 의도하지 않은 요청을, 사용자의 권한으로 실행**시키는 공격.

**방어 방법**

- **CSRF 토큰**: 서버가 페이지에 예측 불가능한 랜덤 토큰을 심고, 요청 시 검증. 쿠키는 자동으로 붙지만 이 토큰은 자동으로 안 붙는 게 핵심.
- **SameSite 쿠키**: `Lax`/`Strict`로 설정 시 다른 사이트발 요청엔 쿠키 미첨부. 최신 브라우저 기본값은 `Lax`.

> Spring Security는 CSRF 보호가 **기본으로 켜져 있어서**, 실습 초반 POST 요청이 403으로 막히는 건 버그가 아니라 정상 동작.
>

## 🔄 세션 로그인 전체 흐름 (6단계)

| 단계 | 주체 | 동작 |
| --- | --- | --- |
| ① 로그인 요청 | 브라우저 | ID / 비밀번호 전송 |
| ② 인증 + 세션 생성 | 서버 | 서버가 세션을 저장소에 보관 |
| ③ Set-Cookie 응답 | 서버 | JSESSIONID 쿠키 전달 |
| ④ 쿠키 자동 저장 | 브라우저 | 브라우저가 보관 |
| ⑤ 이후 모든 요청 | 브라우저 | `Cookie` 헤더 자동 전송 |
| ⑥ 세션 조회 → 인증 확인 | 서버 | 보호 API 접근 허용 |

## 🌱 Spring Security 내부 동작 (심화)

### A. Spring Security의 정체는 "필터 체인"

Spring Security는 마법이 아니라 **컨트롤러 앞에 줄지어 선 서블릿 필터들의 모음**이다.
요청은 `DispatcherServlet`(컨트롤러)에 닿기 **전에** 이 필터 줄(`SecurityFilterChain`)을 통과한다.

| 필터 | 책임 |
| --- | --- |
| `SecurityContextHolderFilter` | 세션에서 인증 정보를 꺼내옴 |
| `UsernamePasswordAuthenticationFilter` | 로그인 폼 요청 가로채 인증 처리 |
| `CsrfFilter` | CSRF 토큰 검증 |
| `LogoutFilter` | 로그아웃 요청 처리 |
| `AuthorizationFilter` | 해당 URL 접근 권한 확인 |

→ 우리가 작성할 `SecurityFilterChain` 빈 설정 = **"이 필터 줄을 어떻게 구성할지"** 정의하는 일.

### B. 인증 정보는 어디에 사는가 — SecurityContext와 ThreadLocal

- 로그인된 사용자 정보 = `Authentication` 객체
- `Authentication` → `SecurityContext`에 담김 → `SecurityContextHolder`가 보관
- `SecurityContextHolder`는 기본적으로 **ThreadLocal** 사용 (= 현재 요청 처리 스레드 안에서만 보이는 저장 공간)
- 인증 정보는 **요청 한 건이 처리되는 동안만** 살아있다가 끝나면 비워짐.

현재 사용자 꺼내기:

```java
SecurityContextHolder.getContext().getAuthentication()
```

> 의문: 요청 끝나면 비워지는데 다음 요청 땐 어떻게 다시 알아볼까? → **세션**이 답.
> 

### C. 쿠키 → 세션 → 인증 정보 연결고리

```
[브라우저]                          [서버]
JSESSIONID 쿠키  ──요청 시 자동 전송──▶  JSESSIONID로 세션 조회
                                         └─ HttpSession (세션 ID와 1:1 매핑)
                                              └─ SecurityContext
                                                   └─ Authentication
                                                        = 인증된 사용자 정보 + 권한
```

이 연결을 담당하는 부품: `SecurityContextHolderFilter` + `HttpSessionSecurityContextRepository`

**요청당 동작 순서**

1. 요청 도착 → 필터가 JSESSIONID로 `HttpSession`을 찾음
2. 세션에 저장돼 있던 `SecurityContext`를 꺼내 `SecurityContextHolder`(ThreadLocal)에 올림
3. 컨트롤러가 일하는 동안 인증 정보 사용 가능
4. 요청 종료 시 ThreadLocal을 비움

> **기억법**: 인증 정보의 영구 보관소 = **세션**, 요청 중 임시 작업대 = **ThreadLocal**, 보관소 여는 열쇠 = **JSESSIONID**
> 

### D. 로그인이 실제로 처리되는 순간

`/login`에 ID/PW를 POST하면:

```
UsernamePasswordAuthenticationFilter   (요청 가로채기)
        ↓
AuthenticationManager                  (총괄 관리자)
        ↓
AuthenticationProvider                 (실제 검증 담당)
        ├─ UserDetailsService          (DB에서 사용자 정보 로드)
        └─ PasswordEncoder             (입력 PW vs 저장된 해시 비교)
```

**실습 직결 핵심 2가지**

1. 비밀번호는 **절대 평문 저장 금지.** 회원가입 시 `PasswordEncoder`(보통 `BCryptPasswordEncoder`)로 해싱 저장, 로그인 시엔 같은 방식으로 비교만. 원본 PW는 서버도 모르는 게 정상.
2. 인증 성공 시 `Authentication`을 `SecurityContext`에 담아 세션에 저장 + **세션 ID 새로 발급**(세션 고정 방어, 기본 활성화).

### E. 로그아웃의 내부 동작

`LogoutFilter`가 **세 가지를 모두** 처리해야 함:

1. 서버의 `HttpSession` 무효화 (`invalidate`) ← **가장 중요**
2. `SecurityContext` 비우기
3. 클라이언트의 JSESSIONID 쿠키 삭제

> 서버 쪽 세션이 살아있으면 쿠키만 지워도 의미 없음. ("상태는 서버가 들고 있다"는 세션 방식의 본질)
> 

### F. CSRF in Spring Security

`CsrfFilter`가 처리:

- GET·HEAD 등 **조회성 요청은 통과**
- POST·PUT·DELETE 등 **상태 변경 요청에만** CSRF 토큰 요구 → 없거나 틀리면 **403**

**실습에서 마주칠 상황**

- Postman/fetch로 POST(로그인·회원가입) 보내면 403 → 버그 아님, CSRF 보호 정상 작동.
- 처리 방식은 아키텍처에 따라 갈림:
    - 서버 렌더링(템플릿) 방식 → 토큰을 폼에 심어 전송
    - REST API + SPA/앱 방식 → 흔히 CSRF를 끄는 대신 다른 방어(SameSite 쿠키, 적절한 CORS)에 의존

## 🧠 전체 그림 한 문장 요약

> 필터 체인이 요청을 받아 → 세션에서 인증 정보를 꺼내 ThreadLocal에 올리고 → 로그인 시엔 Manager·Provider·UserDetailsService·PasswordEncoder가 협력해 검증한 뒤 세션에 저장하고 → 로그아웃 시엔 세션을 무효화한다. CSRF 필터는 그 와중에 위조 요청을 막는다.
