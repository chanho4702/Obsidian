# 3단계. JWT 직접 구현

> [!abstract] 한 줄
> JWT는 `header.payload.signature` 세 조각이다. **payload는 암호화가 아니라 그냥 인코딩**(누구나 읽힌다)이고, signature는 "아무나 못 만든다"를 보장할 뿐 "아무도 못 읽는다"가 아니다. 라이브러리 없이 손으로 한 번 까보는 단계.

실습: `C:\myStudy\jwt_Login` — Boot4 / Thymeleaf. **라이브러리 없이**(`javax.crypto` HmacSHA256 + Base64Url) JWT를 직접 조립/검증하고, **Spring Security 없이** 순수 MVC 인터셉터로 Bearer 검증.

---

## 1. JWT 구조

```
eyJhbGc... . eyJzdWI... . SflKxwRJ...
  Header   .   Payload   .  Signature
```

| 조각 | 내용 | 인코딩 |
|---|---|---|
| **Header** | 알고리즘(`alg: HS256`), 타입(`typ: JWT`) | Base64Url |
| **Payload** | 클레임(`sub`, `role`, `exp`, `jti`...) | Base64Url — **암호화 아님, 그냥 읽힘** |
| **Signature** | `HMAC_SHA256(base64url(header) + "." + base64url(payload), secret)` | — |

> ⚠️ **payload에 비밀번호·민감정보를 넣지 마라.** Base64Url은 누구나 디코드한다(브라우저에서 jwt.io 없이도 읽힘).

## 2. 서명(HMAC)의 의미

- HMAC = **대칭키 서명**. 같은 `secret`을 아는 쪽만 서명을 만들 수 있다.
- 검증 = 받은 `header.payload`로 서명을 다시 계산해 토큰의 signature와 **바이트 비교**.
  - 한 글자라도 위조하면(예: `role: USER → ADMIN`) 서명이 안 맞아 거절된다.
- 서명이 보장하는 것: **"위조 안 됨(무결성)·발급자 인증"** — 기밀성(읽기 차단)이 아니다.

## 3. 만료 (exp)

- payload의 `exp`로 유효시간을 못박는다. 실습에선 **60초**로 짧게 둠 → 만료 동작을 눈으로 보려고.
- 짧은 만료의 부작용("만료될 때마다 다시 로그인?")이 [[4단계. 토큰 보안 (Access·Refresh)|4단계]]의 Refresh Token 떡밥.

## 4. 검증 실패의 종류 (실습 핵심)

| 코드 | 상황 |
|---|---|
| `MALFORMED` | 형식 오류 / 토큰 없음 |
| `BAD_SIGNATURE` | 서명 불일치 = 위조 (payload를 건드림) |
| `EXPIRED` | `exp` 지남 |

화면(Thymeleaf)에서 토큰을 빨강/보라/파랑 3등분해 보여주고, ✅정상 / 🚫토큰없이 / 🕵️위조(role→ADMIN) / ⏰만료를 버튼으로 재현한다.

## 🎯 어디까지 학습해야 하나 (중요)

> 결론: **이 코드를 백지에서 짤 수 있을 만큼 외울 필요 없다.** `JwtProvider`(HMAC 직접 구현)는 "실무 코드"가 아니라 **"JWT 내부를 들여다보는 X-ray"**다. 실무에선 절대 직접 안 짠다 — `jjwt`·Nimbus·Spring Security가 해주며, 보안 코드를 직접 짜는 건 오히려 위험(타이밍 공격·알고리즘 혼동 공격).

**🧠 평생 가져갈 개념 (★진짜 학습 대상)**
- JWT = `header.payload.signature`, payload는 **인코딩일 뿐**(누구나 읽힘 → 비밀번호 금지)
- 서명의 의미: "아무나 못 만든다" ≠ "아무도 못 읽는다"
- **stateless** — 서버가 세션을 안 들고 요청마다 토큰을 재검증 (← [[2단계. 세션의 한계와 JWT의 등장|2단계]]에서 본 확장 문제의 해법)
- 만료가 왜 필요하고, 짧으면 왜 Refresh가 필요해지는가 (→ 4단계)
- 검증 실패 3종: 형식 / 서명불일치 / 만료

**🔧 "할 줄 알면 되는" (외울 필요 X, 라이브러리에 위임)**
- 토큰 발급/검증 코드 자체, Base64Url·`Mac.getInstance(...)` 같은 디테일

**🗑️ 안 외워도 되는 것**
- `sign()`, `MessageDigest.isEqual()` 같은 구현 디테일 — "이렇게 돌아가는구나" 한 번 보면 충분

### 학습 완료 기준 (말로 설명되면 통과)
1. 위조 토큰(role→ADMIN)이 왜 `BAD_SIGNATURE`로 막히는가?
2. 60초 뒤 정상 토큰이 왜 `EXPIRED`가 되는가?
3. 토큰 없이 호출하면 서버는 그걸 어떻게 아는가?

## 설계 결정 요약 (왜 이렇게 짰나)
- **라이브러리 없이 직접** — 3단계 목표가 "구조 그 자체"라 jjwt로 가리지 않음. 이게 **마지막으로 "직접" 짜는 단계** ([[5단계. Spring Security 심화|5단계]]부터 표준 부품에 위임).
- **Spring Security 미사용** — Bearer 검증을 순수 MVC 인터셉터로. Security Filter 이관은 5단계.
- **DB 없이 인메모리** — 관심사는 토큰이지 영속성이 아니라 단순화.
