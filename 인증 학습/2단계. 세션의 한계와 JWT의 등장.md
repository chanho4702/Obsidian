# 2단계. 세션의 한계 — 왜 JWT가 등장했나

> [!abstract] 한 줄
> 세션은 **상태를 서버가 들고 있다.** 서버가 한 대일 땐 문제없지만, 트래픽이 늘어 서버를 여러 대로 늘리는 순간(수평 확장) "그 세션이 어느 서버에 있느냐"가 발목을 잡는다. 이 문제를 푸는 한 갈래가 **세션을 외부 저장소(Redis)로** 빼는 것이고, 다른 갈래가 **상태를 클라이언트에게 넘기는(JWT)** 것이다.

이 단계는 실습보다 **"왜"를 잡는 개념 단계**다. [[1단계. Session Login]]에서 만든 세션 방식의 한계를 이해해야 [[3단계. JWT 직접 구현|JWT]]가 왜 필요한지 납득된다.

---

## 1. 출발점 — 세션은 서버 메모리에 산다

[[1단계. Session Login]]에서 본 것: 로그인하면 서버가 세션을 만들어 **자기 메모리**에 보관하고, 클라이언트엔 열쇠(`JSESSIONID`)만 준다. 서버가 한 대면 완벽하다. 문제는 **서버가 여러 대**가 될 때 생긴다.

## 2. 수평 확장이 깨뜨리는 지점

트래픽이 늘면 서버를 여러 대(인스턴스)로 늘리고 앞에 로드밸런서를 둔다. 그런데:

- 로그인 요청은 **A 서버**가 받아 세션을 A의 메모리에 만들었다.
- 다음 요청이 로드밸런서에 의해 **B 서버**로 갔다.
- B는 그 세션을 모른다 → **"누구세요?" = 로그아웃 취급.**

즉 세션은 **특정 서버에 묶인 상태(stateful)**라 인스턴스를 자유롭게 늘릴 수 없다.

## 3. 임시방편과 그 대가

| 해법 | 방식 | 한계 |
|---|---|---|
| **Sticky Session** | 로드밸런서가 같은 사용자를 항상 같은 서버로 고정 | 그 서버가 죽으면 세션 전부 소실, 부하 불균형, 무중단 배포 곤란 |
| **Session Clustering** | 서버끼리 세션을 복제·동기화 | 서버 수에 비례해 복제 비용 폭증, 확장성 나쁨 |
| **외부 세션 저장소 (Redis Session)** | 세션을 모든 서버가 공유하는 Redis에 저장 | 실무에서 널리 쓰는 정답 중 하나. 단, 요청마다 Redis 조회(네트워크 왕복)·Redis가 새 단일 장애점 |

> Sticky/Clustering은 근본 해결이 아니라 회피다. **Redis Session**은 진짜 해법이고, [[4단계. 토큰 보안 (Access·Refresh)|4단계]]에서 실제로 Redis를 쓴다.

## 4. 다른 갈래 — 상태를 클라이언트에게 넘긴다 (JWT)

발상의 전환: 서버가 상태를 안 들면 어느 서버로 가든 상관없어진다. 그래서 **인증된 사용자 정보 자체를 토큰에 담아 클라이언트가 들고 다니게** 한다.

- 요청마다 클라이언트가 토큰을 보내고, 서버는 **서명만 검증**하면 끝 → 서버는 아무 상태도 저장하지 않는다(**stateless**).
- 어느 인스턴스로 요청이 가도 동일하게 검증 가능 → 수평 확장 자유.

이것이 [[3단계. JWT 직접 구현|JWT]]가 등장한 핵심 이유다.

## 5. 공짜는 아니다 — JWT의 트레이드오프

상태를 클라이언트에 넘기면 **"서버가 즉시 무효화할 수 없다"**는 새 문제가 생긴다. 발급한 토큰은 만료 전까지 유효하다 — 강제 로그아웃·탈취 대응이 어렵다. 이 빈틈을 메우려고 다시 **짧은 만료 + Refresh Token + 블랙리스트(Redis)**가 등장한다 → [[4단계. 토큰 보안 (Access·Refresh)|4단계]].

> 결국 "순수 stateless"는 환상이고, 실무는 **stateless를 기본으로 하되 필요한 곳만 stateful(블랙리스트)**로 절충한다.

---

## 핵심 한 줄 정리

> 세션은 서버에 상태를 묶어 **수평 확장을 막는다.** 이를 푸는 두 갈래 — 세션을 **Redis로 공유**하거나, 상태를 **JWT로 클라이언트에 넘긴다.** JWT는 확장은 자유롭지만 **즉시 무효화가 어렵다**는 새 숙제를 남긴다(→ 4단계).
