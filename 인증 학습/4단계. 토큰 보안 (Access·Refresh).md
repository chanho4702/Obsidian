# 🛡️ 4단계. 토큰 보안 — Access / Refresh

> **한 줄 요약**
> "못 막는 건 짧게(AT), 막을 수 있는 건 서버가 기억(RT)."
> 흔한 검증은 빠르게(stateless), 갱신·무효화는 안전하게 가끔만(stateful) 처리하는 설계.

---

## 1. 왜 토큰을 둘로 나누나

- **AT 단독의 한계** = *한 번 발급하면 만료까지 못 막는다* → 그래서 수명을 **짧게**.
- 짧게 하면 5분마다 로그인? → UX 박살 → **RT로 조용히 재발급(silent refresh)**.
- **부가효과 = 속도 + 확장성** (아래 핵심).

### 왜 속도까지 챙기나 (★ 4단계 핵심 통찰)

모든 보호된 요청은 "너 인증된 사람 맞아?"를 검사해야 한다. 방법은 둘.

|        | Stateful (서버가 기억)      | Stateless (토큰이 자체 증명)   |
| ------ | ----------------------- | ---------------------- |
| 방식     | 세션ID/토큰을 DB·Redis에서 조회   | 서명을 그 자리에서 다시 계산(HMAC) |
| 비용     | 네트워크 + 저장소 I/O (느림, ms)  | CPU 연산만 (초고속, µs)       |
| 확장     | 모든 서버가 같은 저장소 공유 필요      | 아무 서버나 혼자 검증 가능         |
| 무효화    | ✅ 쉬움 (지우면 끝)             | ❌ 어려움 (만료까지 못 막음)       |

**딜레마**: 순수 Stateful(옛 세션)은 매 요청 저장소를 두드려 병목(→ 2단계 "세션의 한계"). 순수 Stateless(JWT만)는 빠르지만 무효화 불가.

**해법 = "검사 빈도"로 역할 분담**

```
AT (Stateless) → 매 요청마다 검증   → I/O 0번, CPU만   → '뜨거운 경로'(빠름 필요)
RT (Stateful)  → 만료될 때만 검증   → I/O 있지만 가끔   → '차가운 경로'(안전 필요)
```

**숫자 체감** (1초 1요청 × 30분 = 1,800요청, AT 수명 15분 가정):

|          | 순수 세션 방식    | AT/RT 분리         |
| -------- | ----------- | ---------------- |
| Redis 조회 | 1,800번(매 요청) | 2번(15분마다 refresh) |
| 나머지      | —           | CPU 서명검증만(I/O 0)  |

→ 저장소 부하 **약 1,000배↓**. AT 검증은 서버 간 공유 불필요 → **수평확장 쉬움**(sticky session·세션 클러스터링 불필요). 2단계의 "세션이 어느 서버에 있나" 문제가 사라짐.

---

## 2. 두 토큰의 역할 분담

|         | **Access Token** | **Refresh Token**  |
| ------- | ---------------- | ------------------ |
| 정체      | JWT (자체검증)        | 불투명 랜덤문자열 (JWT 아님) |
| 서버 기억   | ❌ stateless      | ✅ Redis가 기억        |
| 수명      | 짧다 (30초 / 실무 5~15분) | 길다 (10분 / 실무 수일~수주) |
| 저장 위치   | **JS 메모리 전역상태**   | **HttpOnly 쿠키**    |
| 검사 빈도   | 매 요청             | AT 만료 시에만          |
| 무효화     | jti 블랙리스트 (예외적)   | 키 삭제 (언제든)         |

---

## 3. 4대 보안 장치

1. **AT 짧게** — 탈취돼도 피해가 그 시간으로 제한
2. **RT 회전(Rotation)** — refresh마다 새 RT 발급, 옛 건 폐기
3. **재사용 탐지** — 폐기된 옛 RT가 또 오면 = 탈취 정황 → 패밀리 전체 폭파(강제 로그아웃)
4. **블랙리스트** — 로그아웃 시 jti 등록 → 만료 전 AT도 즉시 `REVOKED`

### 회전 & 재사용 탐지 메커니즘

```
refresh:<tokenId> = "<familyId>|<username>"   (TTL = RT 수명; 회전 후에도 남겨둠)
family:<familyId> = "<현재 유효한 tokenId>"    (이 포인터가 가리키는 토큰만 유효)

로그인       → A 발급, family→A
/refresh(A)  → B 발급, family→B   (A의 키는 안 지움)
/refresh(A)  → family(B) ≠ A → "폐기된 옛 토큰이 또 왔다" = 탈취 의심!
               → 패밀리 폭파(포인터 삭제). 진짜 사용자도 강제 로그아웃되지만 안전 우선.
```

> 정상 사용자는 항상 "가장 최근 RT" 하나만 들고 있다. 옛 RT가 다시 보이면 = 누가 복제한 것.

### 블랙리스트 트레이드오프

- 로그아웃처럼 *예외적인* 토큰만 등록 (전체가 아니라 소수)
- 켜는 순간 그 요청은 다시 stateful (Redis 1회 조회) → **꼭 필요한 토큰만** 올림

---

## 4. 저장 위치 트레이드오프

- AT를 `localStorage`에 → XSS가 읽어감 ❌ → **메모리 변수**에 (새로고침 시 RT로 복구)
- RT를 **HttpOnly 쿠키**에 → JS는 못 읽지만 **브라우저는 자동 전송** → XSS 방어 + 기능 정상
- `SameSite=Strict` (CSRF 방어) + `Path=/api/auth` (노출 최소화)

> "클라이언트가 못 본다"의 정확한 의미 = **JavaScript가 못 읽는다**. 통신은 정상.

## 6. 자가진단

> **"AT는 왜 짧고 stateless로, RT는 왜 길고 stateful로 두는가?"**
> → 흔한 검증은 빠르게/확장가능하게(AT-stateless), 무효화·갱신은 안전하게 가끔만(RT-stateful) 처리하려고.

- [ ] AT/RT를 왜 나누는지 (한계 + 속도 + UX) 설명 가능
- [ ] 회전 + 재사용 탐지 흐름 설명 가능
- [ ] RT를 HttpOnly 쿠키에 두는 이유(XSS) 설명 가능
- [ ] 블랙리스트가 왜 "예외에만" 쓰이는지 설명 가능
